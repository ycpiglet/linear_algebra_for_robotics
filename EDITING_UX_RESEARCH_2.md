## 1. 한글(HWP) — 한국어 감독자의 UX 기대치

이번 조사에서 유일하게 검증까지 완료된 영역이며, 결과가 설계에 직접적으로 유리하다.

- **[검증]** 한/글의 '변경 내용 추적'은 삽입·삭제를 본문과 시각적으로 구분해 표시한다 — Word와 동형의 인라인 변경 표시 관례가 한국어 사용자에게도 형성되어 있다.
- **[검증]** 검토자(reviewer) 단위 필터링을 지원한다 — "검토자별로 변경을 묶어 본다"는 리뷰 단위 기대치. 우리의 `actor:` 라벨별 필터와 정확히 대응.
- **[검증]** 한/글은 **'교정 부호'를 전용 1급 기능으로 제공한다**: 맞춤법·띄어쓰기·오자·탈자·어색한 표현을 바로잡기 위해 문서 임의 위치에 교정 부호를 삽입한다. "지면 위 교정부호" 관례가 한국 워드프로세서에 소프트웨어 기능으로 존재한다.
- **[검증]** 교정 부호는 부호만 삽입하거나 **'추가하거나 바꿀 내용'(제안 대체 텍스트)을 부호와 함께** 삽입할 수 있다 — 교정 표시 1건이 "지적 + 제안 텍스트"를 함께 담는 단위가 될 수 있다는 국내 선례. 우리의 "코멘트 vs 수정 제안" 이원 구조와 정확히 동형이다.
- **[1차출처]** 그 외: 4가지 보기 모드(원본/원본+변경/최종본+변경/최종본), 검토자별 색상, 여백 풍선 + 본문-풍선 연결선 옵션, 개별·일괄 수락/거부.

**설계 시사점**: 원래 사용자가 요청의 출발점으로 삼은 "원고지 교정" 은유가 한국어 도구 생태계에서 이미 소프트웨어 관례로 확립되어 있다. 편집 데스크 UI의 어휘를 여기에 맞춘다 — "수정 제안"을 "교정"으로, 액션 아이콘·용어를 한/글 사용자가 이미 아는 이름으로 붙이면 학습 비용이 0에 가까워진다.

## 2. 공백 영역에서 회수된 패턴 [1차출처]

- **Apple Pages**: 변경 추적을 끄려면 미결 변경을 전부 수락/거부해야 한다(몰래 버리기 불가) — "미결 제안이 남아 있으면 배치를 닫을 수 없다"는 규칙의 선례. 해상도 3계층(변경 1건 / 선택 범위 / 문서 전체)도 제공.
- **Acrobat**: ① 주석을 문서와 분리된 파일(FDF/XFDF)로 직렬화하는 오버레이 데이터 모델 — "주석 레이어 별도 저장 + 동기화" 아키텍처의 상용 선례. ② PDF 교정 주석을 **원본 Word 문서로 내보내 소스를 직접 수정**하는 파이프라인이 존재하며, 이때 기계 적용 가능한 편집(삽입/삭제/대체)만 선별하고 최종 수락은 인간이 건별로 한다 — 우리의 "주석 → .qmd 자동 반영 + 코멘트 유형 분류 + 승인 게이트"와 동형의 상용 선례.
- **Figma**: 공간 앵커 댓글 핀(호버 미리보기·클릭 확장), C 단축키로 여는 댓글 사이드바, resolve 시 캔버스·사이드바에서 숨김 + "Show resolved" 토글 — 그림·다이어그램 위 주석에 이식할 패턴(우리 교재의 SVG 그림 주석에 적용 가능).
- **GitBook change request**: 리뷰 단위는 문단이 아니라 스페이스 단위 브랜치(여러 페이지 편집 묶음)이고 diff는 split-view 기본. 단 **병합 후 롤백 불가** — 우리의 revert 가능 모델이 우위인 지점.
- **PubPub**: 커뮤니티 리뷰는 "원고 전체 + 리뷰 기간(예: 8주)"이 묶음 단위이고, 조사된 프로젝트 다수가 주석을 실시간이 아니라 **기간 종료 후 일괄 반영**했다 — 배치 처리(§5.6)의 학술 출판 선례.
- **Gemini in Docs**: 제안마다 개별 수락/거부 컨트롤 제공(부분 수락 가능) — 일괄 적용만 있는 게 아니다.

## 3. 편집 이력의 알갱이 크기 — coalescing 선례 [1차출처]

"같은 문구를 여러 번 고치면 커밋도 여러 개인가?"에 대한 업계의 답은 일관된다: **아니다. 영구 이력의 단위는 기술적 변경 횟수가 아니라 '의미 있는, 통째로 되돌릴 수 있는 행위'다.**

### 3.1 에디터 undo 시스템

- **NSUndoManager(Apple)**: undo 그룹의 설계 기준은 "전체가 한 번에 되돌릴 수 있는(entirely revertible) 의미 있는 행위" 단위. 프레임워크가 이벤트 루프 사이클 단위로 자동 병합해 주며, 개발자가 매 변경마다 이력 단위를 만들지 않는다.
- **Blender 설계 논의**: coalescing 축은 시간 기반 vs 위치·의미 경계 기반이며, 시간 기반은 예측 불가능성(타이핑 속도가 임계 근처면 어떤 편집이 묶일지 사용자가 모름) 때문에 기각되고 **"같은 위치, 끊기지 않은 연속"이라는 편집 국소성 규칙**이 채택됨. 커서 이동이 병합을 끊는다.
- **undo 라이브러리 관례**: FIRST/LAST/중복 제거 등 압축 모드가 코드화되어 있고, 공식 근거는 "변경은 많지만 보존 가치가 있는 것은 첫 상태 또는 끝 상태뿐인 경우".

**이식**: 우리 §5.5의 확정(settle) 규칙에 Blender의 국소성 원칙을 반영 — 같은 앵커에 대한 연속 수정은 병합하되, **다른 앵커로 개입 대상이 이동하면(=커서 이동에 해당) 그 시점 상태로 확정**한다. 시간 임계값 기반 자동 확정은 예측 불가능성 때문에 쓰지 않는다.

### 3.2 Git 규범 — 그리고 기존 설계의 수정 필요 지점

- **리베이스의 황금률(Atlassian)**: 공개 브랜치에는 rebase 금지. **PR을 생성하는 순간 브랜치는 '공개'가 되므로**, 이력 다듬기는 원칙적으로 PR 제출 전에 한다.
- **force-push와 리뷰 앵커(GitHub)**: force-push하면 기존 리뷰 코멘트가 코드 문맥과 어긋나 'outdated' 처리된다 — 리뷰 중인 브랜치의 이력 재작성은 리뷰어의 작업을 파괴한다.
- **thoughtbot 프로토콜**: 리뷰가 끝난 뒤, 병합 직전에 1회 인터랙티브 rebase(autosquash)로 정리하는 것이 명시적 팀 규범. `git commit --fixup` + `rebase --autosquash`가 도구 지원. 기능 브랜치는 병합 전 보통 1~3개 커밋으로 압축된다.
- **병합 전략(Graphite)**: squash merge는 개별 커밋의 작성자 정보까지 잃는다. merge commit이 기여 추적 투명성이 가장 높다.

**⚠ 기존 설계 수정**: 기획서 §5.5 초안은 "배치 PR이 열려 있는 동안 같은 앵커의 연속 수정을 amend/fixup으로 합친다"고 했는데, amend는 force-push를 유발해 열린 PR의 리뷰 코멘트 앵커를 파괴한다. 수정된 규칙:

1. **PR 수명 중**: 같은 앵커의 후속 수정은 `fixup!` 커밋으로 **덧붙인다**(append-only, force-push 없음 → 리뷰 앵커 안전).
2. **병합 요청 시점**: autosquash 1회로 fixup들을 대상 커밋에 흡수(단 한 번의 이력 정리, thoughtbot 규범과 일치). 이 시점은 리뷰가 끝난 뒤라 앵커 파괴 비용이 없다.
3. **병합은 merge commit으로**: squash merge는 배치 안의 교정별 커밋과 작성자 구분(R3)을 소멸시키므로 배치 PR에 사용 금지.

### 3.3 버전 기록 UX

- **Google Docs**: 버전 기록을 시간 블록으로 자동 압축해 보여주되 펼치면 분 단위까지 보이고, 사용자가 '이름 있는 버전(named version)'으로 마일스톤을 영구 표시할 수 있다 — "자동 압축 + 명시적 마일스톤"의 2층 구조.

**이식**: 감독자용 이력 뷰도 동일 구조로 — 기본은 배치(PR) 단위로 압축해 보여주고, 펼치면 교정별 커밋이 보이며, 원고의 의미 있는 시점(장 완성, 감수 완료)은 Git 태그를 '이름 있는 버전'으로 노출한다.

### 3.4 AI 학습 관점의 판단 (d)

선례들의 공통 논리 — thoughtbot: "미래의 git blame 사용자는 '사소한 실수를 곧바로 고친 커밋'이 아니라 '의미 있는 변경을 설명하는 커밋'을 봐야 한다" — 는 스타일 학습에도 그대로 적용된다. **다듬는 중간 상태는 압축하고 확정 상태만 학습 원료로 쓴다**는 §5.5의 판단을 유지한다. 단, 협의형 스레드의 "검토 후 버린 대안"은 중간 상태가 아니라 명시적 판단 기록이므로 보존한다(§7.3의 신호 위계 유지).

## 4. 남은 공백과 후속 조치

이번 실행에서 세션 한도로 유실되어 **미조사로 남은 것**: Dependabot/Renovate grouped updates의 구체 설정·도입 배경, changesets 모델, stacked PR 철학, webhook vs 폴링 vs digest의 정량 트레이드오프, Typst 웹앱·Confluence 리뷰 단위, Notion AI 상호작용 디테일, .qmd 블록 ID Lua 필터 실전 사례, Hypothesis↔GitHub 동기화 사례, 위키백과 리비전 모델(minor edit·rollback·봇 노이즈), 이벤트 소싱/CRDT 이력 압축.

두 조사 모두 resume 가능한 상태로 저장되어 있어(완료된 에이전트는 캐시 재생), 사용 한도 리셋 후 재실행하면 실패한 검증·수집만 이어서 돌릴 수 있다. 다만 §3의 Git 규범(황금률, autosquash, 병합 전략)은 널리 알려진 표준 관행이라 실무 채택에 충분하고, 실제로 설계 수정(§3.2)에 이미 반영했다.
