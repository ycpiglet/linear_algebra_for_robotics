> 문서 성격: `AUTHORING_MANUAL.md`(최상위 편집 계약)를 보조하는 편집 도구·워크플로우 설계서
>
> 작성일: 2026-07-17
>
> 상태: 제안 — 감독자 승인 후 단계별 구현. 승인된 항목은 `AUTHORING_MANUAL.md` §20 절차에 따라 매뉴얼로 승격한다.
>
> 보조 문서: `EDITING_UX_RESEARCH.md` — 워드프로세서·협업 플랫폼·출판 도구의 리뷰 UX 딥 리서치(검증된 기능 카탈로그, 객체 단위 편집 베스트 프랙티스, 협의형 코멘트·AI 응답 루프, 우선순위 권고)

이 문서는 "감독자와 편집자가 원고지에 교정 기호를 달듯 개입하고, 그 기록이 전부 남아 원복 가능하며, AI가 그 기록에서 편집 기준을 학습해 나머지 분량에 같은 기준을 적용하는" 시스템을 설계한다.

## 1. 문제 정의

### 1.1 현재의 불편

1. **위치를 말로 설명해야 한다.** "몇 페이지 몇 줄의 어떤 표현을 이렇게 바꿔 달라"를 매번 문장으로 명시해야 한다. 이것은 좌표(페이지·줄) 기반 포인터가 리플로우 문서에서 불안정하기 때문에 생기는 앵커링(anchoring) 문제다.
2. **렌더된 결과물 위에서 개입할 수 없다.** 감독자는 완성된 웹 페이지를 읽지만, 수정 요청은 소스 파일(`.qmd`)의 세계에서 이루어진다. 두 세계 사이의 번역을 지금은 사람이 한다.
3. **개입 주체가 구분되지 않는다.** 감독자, 외부 편집자, AI 에이전트의 수정·코멘트가 형식적으로 구분·표시되지 않는다.
4. **편집 의도가 휘발된다.** 개별 교정("이 표현을 저렇게")은 반영되면 사라진다. 그 뒤에 있는 기준("이런 문체를 원한다")이 축적되지 않아, 같은 지적을 장마다 반복해야 한다.

### 1.2 요구사항 (우선순위 순)

| # | 요구 | 판단 기준 |
|---|---|---|
| R1 | 사람이 렌더된 페이지에서 드래그·형광펜·코멘트로 직접 개입 | 파일명·줄 번호를 몰라도 교정 가능 |
| R2 | 사람이 본문을 직접 수정할 수 있는 자유도 | 브라우저에서 벗어나지 않고 문장 교체 가능 |
| R3 | 모든 개입의 주체(사람·에이전트·모델)가 구분되어 기록 | 이력에서 "누가, 언제, 무엇을, 왜"가 복원됨 |
| R4 | 편집 기록의 저장·삭제·원복 | 어떤 편집이든 되돌릴 수 있고, 되돌린 사실도 남음 |
| R5 | AI가 편집 기록에서 스타일·의도·맥락을 학습해 나머지 분량에 동일 기준 적용 | 같은 유형의 지적이 두 번 반복되지 않음 |
| R6 | 안정적 포인터: 문단·절 단위 링크, 출력물 간(웹·PDF·EPUB) 위치 대응 | 특정 문장을 가리키는 링크를 복사·공유 가능 |

### 1.3 설계 원칙

1. **감독자는 렌더된 책 위에서 일한다.** 소스 파일, 경로, 줄 번호, Git 명령은 사람의 인터페이스에서 보이지 않아야 한다. 그 번역은 시스템(에이전트)의 일이다.
2. **앵커는 좌표가 아니라 텍스트다.** 드래그한 인용문 자체가 위치 지정자다. 인용문 + 앞뒤 문맥이면 원고가 수정되어도 위치를 복원할 수 있다.
3. **원본은 계속 `.qmd` + Git이다.** 편집 이력·작성자·원복은 이미 Git이 완벽하게 제공한다. 새로 만들지 않고, 사람이 Git을 몰라도 그 혜택을 받도록 껍데기를 씌운다.
4. **반영은 전부 PR(pull request)을 통과한다.** diff가 곧 교정지이고, 병합이 곧 승인이며, revert가 곧 원복이다. 에이전트든 사람이든 예외 없다. (감독자의 소규모 직접 수정은 §5.2의 완화 규칙을 따른다.)
5. **학습은 암묵적 모델 튜닝이 아니라 명시적 규칙 승격이다.** AI가 편집 기록에서 발견한 패턴은 사람이 읽고 승인할 수 있는 문서(스타일 프로파일 → `AUTHORING_MANUAL.md`)로 증류한다. 감사 가능하고, 원복 가능하고, 어떤 모델·에이전트로 갈아타도 유지된다.
6. **WYSIWYG 전면 이식은 하지 않는다.** 근거는 §8.

## 2. 전체 아키텍처

```text
[레이어 3] 편집 메모리·학습 루프
    편집 이벤트 기록(platform/editorial/) → 스타일 프로파일 → 매뉴얼 승격 → 전 장 일괄 적용 → 자동 검사화
          ▲                                                            │
          │ 기록                                                        │ 기준 적용
          │                                                            ▼
[레이어 2] 편집 데스크 (렌더된 웹 = 작업면)
    형광펜·코멘트(주석 레이어) · "수정 제안" 패널 · 직접 편집 링크 · 리뷰 대시보드
          ▲                                                            │
          │ 렌더                                                        │ 앵커 → 소스 매핑 (에이전트)
          │                                                            ▼
[레이어 1] 단일 원본 (기존 그대로)
    .qmd + Git + PR — 이력·작성자·원복·검증(make validate/test)
```

- **레이어 1**은 이미 완성되어 있다. 손대지 않는다.
- **레이어 2**가 이번 기획의 본체다. 사람의 개입 지점을 전부 렌더된 웹 위로 올린다.
- **레이어 3**이 R5의 답이다. 레이어 2에서 생기는 모든 개입을 구조화 기록으로 남기고, 거기서 기준을 증류한다.

## 3. 사용자 시나리오 (완성 후의 하루)

### 3.1 감독자

1. 배포된 웹 교재를 읽다가 어색한 문장을 드래그한다.
2. 뜨는 팝업에서 **[형광펜] [코멘트] [수정 제안]** 중 하나를 고른다.
   - 코멘트: "여기 '갱신'이 두 가지 뜻으로 쓰였음. 측정 갱신으로 특정할 것" — 끝. 위치 특정은 시스템이 한다.
   - 수정 제안: 선택된 원문이 채워진 입력창에 고친 문장을 직접 타이핑한다.
3. 다음 날, 리뷰 대시보드에서 자신의 코멘트가 PR로 바뀐 것을 본다. diff(교정지)를 훑고 병합 버튼을 누른다. 마음에 안 들면 PR에 한 줄 답글을 달거나 닫는다(= 기각, 기록에 남음).
4. 이전에 병합했던 편집이 마음에 안 들면 대시보드의 편집 이력에서 **[이 편집 되돌리기]** — revert PR이 만들어진다.

### 3.2 외부 편집자

- 감독자와 동일한 UI. 차이는 권한뿐이다: 코멘트·수정 제안은 자유, 병합은 감독자(또는 위임받은 편집자)만 한다. 모든 개입에 계정 이름이 붙는다.

### 3.3 AI 에이전트

1. 새 주석·이슈를 주기적으로 수거해 인용문 앵커로 소스 위치를 찾고, 수정 PR을 만들고, 원 코멘트에 답글로 연결한다.
2. 병합·기각·수정될 때마다 편집 이벤트 레코드를 적재한다(§6).
3. 주기 작업으로 레코드를 분석해 스타일 프로파일 갱신안을 PR로 제안한다(§7).
4. 승격된 규칙으로 아직 감사받지 않은 장을 일괄 점검하고 장별 PR을 낸다.

## 4. 단계별 로드맵

### Phase 0 — 리뷰 기반 마련 (설정 수준, 코드 거의 없음)

| 작업 | 내용 | 수용 기준 |
|---|---|---|
| 0-1 웹 자동 배포 | GitHub Actions로 `make web` → GitHub Pages 배포. `main` 병합마다 갱신 | 리뷰 대상이 항상 최신 URL로 존재 |
| 0-2 PR 미리보기 | PR마다 미리보기 빌드(아티팩트 또는 경로별 배포) | 병합 전에 렌더 결과를 보고 승인 가능 |
| 0-3 주석 레이어 켜기 | 리뷰 빌드에 Hypothesis 임베드(`comments: hypothesis`) + 편집팀 비공개 그룹 | 드래그→형광펜→코멘트→작성자 표시가 즉시 동작 |
| 0-4 텍스트 조각 링크 | 드래그 후 "이 문장 링크 복사" 기능(텍스트 프래그먼트 `#:~:text=`) — `atlas.js`에 소규모 추가 | 특정 문장으로 바로 가는 링크를 대화·이슈에 붙일 수 있음 |

Hypothesis는 R1(형광펜·코멘트·작성자 구분)을 개발 없이 즉시 충족하는 가장 빠른 경로다. 주석 데이터가 외부 서비스에 있다는 한계는 Phase 1의 자체 패널이 성숙하면 자연히 비중이 줄어든다. 리더용 공개 빌드에는 넣지 않고 리뷰 빌드(프로필 분기)에만 켠다.

- 구현됨: `_quarto-review.yml`(리뷰 프로필 — `--profile web,review`로 웹 설정 위에 델타만 얹음, Hypothesis 활성), `assets/includes/review-scripts.html`(드래그 → "이 문장 링크 복사" 텍스트 프래그먼트 버튼), `make review` 타깃, `.github/workflows/publish-web.yml`(main → gh-pages 루트 + `/review/`, PR → `/preview/pr-N/`).
- 배포 구조: 독자용(주석 없음)은 루트, 교정용(Hypothesis)은 `/review/`. PR 미리보기는 일회성 — 다음 main 배포가 지울 수 있으므로 지속 교정은 `/review/`에서 한다.

- 첫 배포 후 확인할 것: `/review/`에서 Hypothesis 사이드바가 뜨는지(프로필 병합으로 `website.comments`가 적용되는지), 문장 드래그 시 링크 복사 버튼 동작, 독자용 루트에는 둘 다 없어야 함.

### Phase 1 — 편집 데스크 v1: GitHub을 백엔드로

| 작업 | 내용 | 수용 기준 |
|---|---|---|
| 1-1 문단 앵커 필터 | Lua 필터로 모든 문단·블록에 안정 ID 부여(내용 해시 기반). 웹·EPUB 공통, PDF는 절 번호로 대응 | 문단 단위 고유 링크 존재, 원고 수정에도 대부분 유지 |
| 1-2 "수정 제안" 패널 | `atlas.js` 확장: 드래그 → 팝업 → 코멘트/수정안 입력 → **GitHub 이슈 자동 생성**(제목·본문·라벨·앵커 메타데이터 자동 충전) | 사람은 파일명·줄 번호 없이 제안 완료. 이슈에 인용문+앞뒤 문맥+페이지 URL+문단 ID가 기계가독 형식으로 담김 |
| 1-3 소스 매핑 브리지 | 에이전트가 이슈의 앵커로 `.qmd` 위치를 찾아 수정 PR 생성, 이슈에 상호 링크 | 코멘트 → PR 전환이 사람 개입 없이 24시간 내 |
| 1-4 리뷰 대시보드 | 사이트 내 한 페이지에서 열린 제안·진행 상태·최근 병합 이력 표시(GitHub API 조회) | 감독자가 GitHub을 열지 않고도 현황 파악 |
| 1-5 주체 구분 규약 | 커밋 author·PR 라벨 규약: `actor:supervisor` / `actor:editor` / `actor:agent`, 에이전트는 전용 author 서명 | 이력에서 사람·에이전트 개입이 기계적으로 분리 가능 |

이슈 생성은 GitHub의 URL 프리필(`/issues/new?title=…&body=…`) 방식으로 시작한다. 별도 서버·OAuth 없이 편집자의 GitHub 로그인 세션을 그대로 쓰므로 구축 비용이 0에 가깝고, 권한·계정 관리도 저장소 권한을 재사용한다.

HTML·EPUB 공통 적용(PDF 제외). Pandoc 3.9로 스모크 테스트 통과.

Chromium 통합 테스트(선택→툴바→다이얼로그→이슈 URL 생성) 통과.
- 남은 수동 설정: 저장소에 `editorial` 라벨 생성(없으면 프리필에서 라벨만 조용히 누락된다).

## 5. 편집 권한과 원복 규약

### 5.1 주체와 권한

| 주체 | 식별 | 할 수 있는 것 | 반영 경로 |
|---|---|---|---|
| 감독자 | GitHub 계정 + `actor:supervisor` | 코멘트·수정 제안·직접 수정·병합·기각·원복 | PR 병합(소규모 직접 수정은 §5.2) |
| 편집자 | GitHub 계정 + `actor:editor` | 코멘트·수정 제안·직접 수정(브랜치) | PR 제출 → 감독자 병합 |
| 에이전트 | 전용 author 서명 + `actor:agent` | 제안 소비·수정 PR·기록 적재·프로파일 제안 | PR 제출 → 감독자 병합. **직접 병합 금지** |

### 5.2 감독자 직접 수정의 완화 규칙

오탈자·조사 수정처럼 명백한 교정까지 PR을 강제하면 편의성이 죽는다. 감독자 본인의 `main` 직접 커밋은 허용하되, 에이전트가 후속으로 편집 이벤트 레코드를 자동 적재해 기록 누락을 막는다. 의미가 바뀌는 수정은 감독자라도 PR을 권장한다(미리보기 검증을 위해).

### 5.3 원복

- 모든 원복은 `git revert`(역방향 커밋)로 한다. 이력을 지우는 force-push는 금지 — "되돌린 사실"도 편집 기록이다.
- 대시보드의 [이 편집 되돌리기]는 revert PR 생성으로 구현한다.
- 주석·이슈의 삭제는 "기각(닫힘)"으로 대신한다. 기각 사유는 학습 신호로 특히 가치가 높다(§7.2).

### 5.4 반영 단위: 커밋과 PR의 역할 분리

"수정 1건 = PR 1개"를 강제하지 않는다. 두 단위의 역할이 다르기 때문이다.

- **커밋 = 기록 단위 (원자).** 교정 1건 = 커밋 1개를 유지한다. 원복(revert), 편집 이벤트 레코드(§6), 객체 단위 이력 조회가 모두 커밋에 붙으므로, 묶어서 병합하더라도 개별 교정을 선택적으로 되돌릴 수 있다. 단 **렌더링 결과에 영향 없는 표면 정리(공백·마크업 정규화 등)는 단독 커밋으로 만들지 않고 실질 교정 커밋에 동승시킨다** — 위키백과 봇 정책(WP:COSMETICBOT)이 대규모 협업에서 확립한 노이즈 방지 규범이다.
- **PR = 검토 단위 (묶음).** 감독자가 한 번에 훑기 좋은 크기로 묶는다. GitHub은 열린 PR에 커밋을 계속 누적할 수 있으므로, PR을 "교정 묶음 컨테이너"로 쓴다.

| 수정 유형 | 반영 경로 | 근거 |
|---|---|---|
| 오탈자·표기 통일 등 명백한 교정 | **장별 누적 배치 PR** — 장마다 열려 있는 "교정 묶음" PR에 커밋을 쌓고, 감독자가 리뷰 세션에서 한 번에 병합 | 개별 PR은 리뷰 피로(노이즈)만 만든다 |
| 의미가 바뀌는 수정, 협의형 왕복의 결과물 | **개별 PR** | 단독 판단이 필요한 변경을 묶음에 섞으면 배치 전체가 막힌다 |
| 규칙 일괄 적용(§7) 같은 대용량 작업 | **장 단위 분할 PR** (§7.4 유지) | 거대 diff는 검토 불가능 |
| 감독자 본인의 소규모 직접 수정 | 직접 커밋 (§5.2 유지) | — |

배치 PR의 운영 규칙: 오래 열어 두면 충돌 위험이 커지므로 **1~3일 또는 커밋 10~15개 중 먼저 도달하는 시점에 병합을 요청**하고, 병합 후 새 배치를 연다. **그 배치에 딸린 미결 제안·코멘트가 남아 있으면 병합을 요청할 수 없다** — 전부 수락·기각·이월 중 하나로 명시 처리해야 한다(Pages가 미결 변경을 전부 해소하기 전에는 변경 추적을 끌 수 없게 강제하는 것과 같은 원리 — 몰래 버려지는 제안이 없게 한다). 배치 안에서 감독자가 특정 커밋만 거부하고 싶으면 해당 커밋을 revert 커밋으로 상쇄한 뒤 병합한다(기각 기록도 남는다). **배치 PR의 병합은 merge commit으로 한다** — GitHub squash merge는 배치 안의 교정별 커밋과 작성자 정보를 하나로 뭉개므로 주체 구분(R3)·선택적 원복과 양립하지 않는다.

### 5.5 이력의 두 층위: 병합 전에는 다듬고, 병합 후에는 덧쓴다

"교정 1건 = 커밋 1개"(§5.4)의 '1건'은 키 입력이나 시도 횟수가 아니라 **확정된 논리적 교정 단위**다. 같은 문구를 여러 번 고쳐 다듬는 과정을 매번 커밋으로 남기지 않는다.

- **병합 전(작업 이력, 다듬기 허용).** 배치 PR이 열려 있는 동안 같은 앵커에 대한 후속 수정은 `fixup!` 커밋으로 **덧붙이고**(append-only — 이미 열린 PR에서 amend/force-push를 하면 리뷰 코멘트 앵커가 outdated로 파괴된다), **병합 요청 시점에 autosquash 1회**로 대상 커밋에 흡수한다. 리뷰가 끝난 뒤의 단 한 번의 이력 정리는 통용되는 Git 팀 규범이다. 다듬는 과정의 중간 상태는 일회성이며 보존 의무가 없다.
- **병합 후(공식 이력, 불변).** 병합된 커밋은 절대 고치지 않는다. 병합 후의 재수정은 새 커밋이고, 되돌림은 revert다(§5.3). 병합이 곧 이력의 공개 경계다.
- **확정(settle) 시점**: ① 배치 PR 병합을 요청할 때 ② 다른 주체가 같은 앵커에 개입할 때(그 시점의 상태로 고정 후 §11-3 적용) ③ **개입 대상이 다른 앵커로 이동할 때**(Yjs가 연속 타이핑을 단일 Item으로 병합하되 같은 주체의 끊기지 않은 연속에만 적용하고 흐름이 끊기면 분할하는 것과 같은 편집 국소성 원칙) ④ 감독자가 명시적으로 확정할 때. 시간 임계값 기반 자동 확정은 어떤 편집이 묶일지 예측 불가능해지므로 쓰지 않는다.
- **편집 이벤트 레코드(§6)도 확정 상태만 적재한다.** 다듬는 중간 타이핑 흔적은 학습 가치가 낮다. 반면 협의형 스레드에서 검토·기각된 대안들은 숙고의 기록이므로 스레드 링크로 보존한다 — 중간 상태와 숙고 기록을 구분하는 것이 원칙이다.

### 5.6 처리 주기와 자동화 비용

실시간 감시는 필요 없다. 교정은 본질적으로 비동기 작업이고, 감독자의 리뷰도 세션 단위로 일어난다.

- **트리거는 다이제스트 주기로.** 에이전트는 상시 대기하지 않고 주기적으로(예: 하루 1~2회, 필요 시 GitHub webhook 병행) 새 코멘트·이슈를 수거해 일괄 처리한다.
- **LLM을 깨우기 전 저비용 사전 필터.** "새 항목이 있는가"는 API 카운트 조회(토큰 소모 0)로 판별하고, 있을 때만 에이전트를 기동한다. 빈 주기에는 비용이 발생하지 않는다.
- **배치 처리가 토큰도 절약한다.** 코멘트 10건을 1회 실행으로 처리하면 장 원문·매뉴얼 규칙 등 컨텍스트 로드를 공유하므로, 10회 개별 실행보다 싸다. 반영 단위(§5.4)와 처리 주기는 이렇게 같은 방향을 가리킨다.
- **긴급 경로.** 감독자가 즉시 반영을 원하면 코멘트에 `지금` 라벨(또는 정해진 키워드)을 달아 다음 주기를 기다리지 않고 처리하게 한다.

## 7. 스타일 학습 루프

### 7.1 왜 "규칙 승격"인가

편집 기록으로 모델을 암묵적으로 학습시키는 대신, **사람이 읽을 수 있는 규칙으로 증류해 문서에 승격**한다.

- 감사 가능: 감독자가 "AI가 내 스타일을 뭐라고 이해했는지"를 문장으로 읽고 교정할 수 있다.
- 원복 가능: 규칙도 Git 이력 안에 있다(R4의 일관 적용).
- 이식 가능: 에이전트·모델을 교체해도 기준이 유지된다. 이 저장소는 이미 `AUTHORING_MANUAL.md`라는 최상위 편집 계약을 갖고 있으므로, 학습의 종착지는 새 시스템이 아니라 **기존 매뉴얼의 자동 성장**이다.

### 7.2 파이프라인

```text
편집 이벤트(JSONL, §6)
  → [주기 분석] 범주별 군집화, 반복 패턴 탐지 (같은 유형 지적 3회 이상 = 규칙 후보)
  → style-profile.md 갱신안 PR   ("관찰: … / 제안 규칙: … / 근거 이벤트: id, id, id")
  → 감독자 리뷰 (승인 / 수정 / 기각)
  → 승인분을 AUTHORING_MANUAL.md 해당 절로 승격 (§20 변경 관리 절차)
  → 미감사 장 일괄 점검 PR + 기계 판정 가능 규칙은 make validate 린트로 추가
```

`platform/editorial/style-profile.md`는 매뉴얼과 달리 "아직 승격되지 않은 관찰"의 완충 지대다. 확신도(관찰 횟수, 기각 이력)와 근거 이벤트 링크를 함께 기록해, 감독자가 근거를 역추적하며 승인할 수 있게 한다.

### 7.3 학습 신호의 위계

| 신호 | 강도 | 예 |
|---|---|---|
| 감독자가 직접 고친 diff | 최상 | before/after 쌍 자체가 정답 레이블 |
| 감독자의 코멘트 원문 | 상 | 의도·어조·판단 기준의 표현 |
| 에이전트 제안의 기각·수정 | 상 | "이렇게 하지 마라"의 명시적 경계 |
| 편집자 제안의 병합 여부 | 중 | 감독자 기준의 간접 관찰 |
| 형광펜만 있고 코멘트 없음 | 하 | 주의 표시로만 취급, 규칙화하지 않음 |

### 7.4 적용 시의 안전장치

- 일괄 적용 PR은 장 단위로 쪼갠다. 거대 diff는 검토 불가능하고, 검토 불가능한 diff는 이 시스템의 목적(사람의 통제)을 배반한다.
- 수학적 의미가 바뀔 수 있는 수정(수식, 정의, 조건)은 일괄 적용에서 제외하고 개별 제안으로 올린다 — 매뉴얼 §17.5의 "기계적 일괄 번역 금지" 원칙을 상속한다.
- 규칙 간 충돌이 감지되면(예: 새 관찰이 매뉴얼 §4.3 표와 상충) 자동 적용하지 않고 충돌 보고 이슈를 만든다.

## 8. 검토한 대안과 기각 이유

| 대안 | 판단 | 이유 |
|---|---|---|
| Google Docs / Notion으로 원고 이관 | 기각 | 코멘트 UX는 최상이나 수식·시맨틱 블록·3중 출력(웹/PDF/EPUB) 파이프라인 전부 상실. 이력이 플랫폼에 종속 |
| 전용 웹앱(WYSIWYG + CRDT 실시간 공동편집) 자체 구축 | 기각 | 구축·유지 비용이 교재 집필을 잠식. 요구사항(R1~R6)은 기존 인프라 조합으로 충족 가능. 실시간 동시편집은 실제 워크플로우(비동기 교정)에 필요 없음 |
| CriticMarkup 인라인 교정 기호 | 보류 | 소스 안 교정 표시는 기계 친화적이나, 사람이 마크업을 타이핑해야 해 R1(편의성)에 역행. 에이전트 간 교정 전달 포맷으로는 추후 유용할 수 있음 |
| PR 리뷰만으로 해결 | 부분 채택 | 반영·승인·이력의 중추로는 채택. 그러나 소스를 보며 리뷰해야 하므로 감독자의 1차 개입 지점으로는 부적합 — 렌더 위 주석 레이어가 그 앞단을 맡는다 |
| Hypothesis 전면 의존 | 부분 채택 | Phase 0의 즉효약으로 채택. 단 주석 원본이 외부 서비스에 있으므로, 에이전트가 이벤트 레코드(§6)로 내재화해 종속을 상쇄. 자체 패널(1-2) 성숙 후 비중 축소 |

**"web으로 이식해야 할까?"에 대한 답:** 이식할 필요가 없다. 매뉴얼 §16.1이 정의했듯 웹은 이미 이 교재의 기준 배포판이다. 필요한 것은 이식이 아니라 그 웹 위에 **개입 레이어**를 얹는 것이고, 원본은 `.qmd` + Git으로 남아야 이력·원복·다중 출력이 유지된다.

## 9. 포인터 규약 (R6)

| 수준 | 웹 | PDF | EPUB |
|---|---|---|---|
| 장·절 | 절 ID 앵커(`anchor-sections`, 기존) + `@sec-` 상호참조 | 책갈피·페이지 번호 | 목차·랜드마크 |
| 문단 | 문단 ID(1-1 필터) 앵커 링크 | 절 번호 + 문단 순번 병기 | 동일 문단 ID |
| 문장·구 | 텍스트 프래그먼트 링크(`#:~:text=`, 0-4) 또는 인용문 앵커 | — (인용문으로 특정) | — (인용문으로 특정) |

교정 대화에서 위치는 항상 **웹 URL + 인용문** 쌍으로 주고받는 것을 표준으로 한다. 페이지·줄 번호는 어떤 워크플로우에서도 요구하지 않는다.

| 위험 | 완화 |

| 스타일 프로파일의 과잉 일반화 | 3회 반복 임계 + 기각 이력 가중 + 승격은 반드시 사람 승인. 자동으로 매뉴얼을 고치지 않음 |
| WYSIWYG 왕복 손상 | 2-2는 대표 장 왕복 무손실 검증 통과 전 도입 금지. 실패 시 github.dev(2-1)만 유지 |

## 11. 시나리오 플레이북

편집 현장에서 반복될 상황과 시스템의 대응을 미리 정한다. 원칙은 §5.4~5.6에서 나오고, 여기는 그 적용례다.

| # | 시나리오 | 대응 | 기록 |
|---|---|---|---|
| 1 | 한 사람이 같은 문구를 여러 번 고쳐 다듬음 | 병합 전이면 fixup으로 쌓고 병합 직전 하나로 흡수(§5.5). 병합 후의 재수정은 새 커밋 | 확정 상태 1건. 병합 후 재수정은 별건 |
| 2 | 수정 → 원복 → 재수정의 흔들림(병합 이후) | 각각 revert·새 커밋으로 정상 처리 | 이벤트 3건 모두 기록하되 스타일 프로파일에는 "미확정 선호" 표시 — 흔들린 주제는 규칙 승격 금지(§7.4) |
| 3 | 에이전트 제안이 미결인데 감독자가 같은 문구를 직접 수정 | 사람이 우선. 에이전트 제안은 자동으로 닫힘 | 상태 `superseded`(기각 `rejected`와 구분 — 내용 거부가 아니라 대상 소멸) |
| 4 | 두 편집자가 같은 문장을 거의 동시에 수정 | 먼저 병합된 쪽이 기준. 나중 것은 리베이스 후 **같은 줄·문단을 건드린 경우에만 충돌로 보고 감독자 판단**(다른 문단이면 자동 병합 — MediaWiki diff3 기준선). 장별 배치 PR로 작업 영역을 나눠 발생 자체를 줄임 | 채택·기각 각각 기록 |
| 5 | 대규모 구조 개편(절 이동·분할·장 재배치) | 구조 변경 커밋과 내용 교정 커밋을 절대 섞지 않음(이동+수정이 한 diff에 있으면 검토 불능). 개편 동안 해당 장 배치 PR 동결, 개편 후 고아 주석 스윕(§10) | 구조 변경은 `category: scope`로 별도 분류 |
| 6 | 협의 스레드가 진행되는 사이 대상 문구가 다른 수정으로 이미 바뀜 | 에이전트가 스레드에 "대상 변경됨"을 알리고 새 문구로 재앵커, 논의 계속 | 스레드 연속성 유지, 앵커 이력 남김 |
| 7 | 같은 유형의 지적이 여러 장에서 반복 | 장마다 개별 처리하지 않고 규칙 후보로 승격(§7.2) 후 일괄 적용 | 이벤트들이 규칙 후보의 근거 목록으로 묶임 |
| 8 | 두 표현 사이에서 결정을 못 내림(A/B) | 두 버전을 각각 미리보기 빌드로 나란히 보고 선택 | 채택안·기각안 쌍으로 기록 — 기각안이 학습 신호(§7.3) |
| 9 | 급한 오류(수식 오타 등)를 즉시 반영해야 함 | `지금` 키워드로 긴급 경로(§5.6), 배치 주기를 건너뜀 | 통상과 동일 |

## 12. 착수 순서 제안

1. **Phase 0 전체** — 설정 수준이며 즉시 체감 효과가 가장 크다. 이번 브랜치에서 바로 구현 가능.
2. **1-1 문단 앵커 필터 + 1-2 수정 제안 패널** — 편의성의 본체.
3. **1-3 브리지 + 3-1 이벤트 적재** — 이 둘이 붙는 순간부터 편집 기록이 자산으로 쌓이기 시작하므로, 직접 편집(Phase 2)보다 먼저 한다. 늦게 시작할수록 학습 원료를 잃는다. ~~착수 전에 기술 스파이크 1회 선행~~ → 스파이크 완료·구현됨(Phase 1 구현 노트 참조).
4. 이후 Phase 2 → 3-2~3-5 순.
