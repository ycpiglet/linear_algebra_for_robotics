# Robotics Math Atlas 영구 저작·편집 매뉴얼

> 문서 성격: 사용자 요구를 보존하는 최상위 편집 계약
>
> 적용일: 2026-07-16
>
> 이번 집필 범위: **자코비안(Jacobian), 최대우도추정(Maximum Likelihood Estimation, MLE), 칼만 필터(Kalman filter, KF)**
>
> 장기 확장 범위: 로봇공학, 인공지능, 역학·기구학, 센서공학, 구동기·장비·모듈. 단, 이번 작업에서는 구조와 예정 항목만 만들고 본문을 집필하지 않는다.

이 문서는 대화에서 드러난 요구와 불편을 잊지 않고, 저자·편집자·개발자가 같은 기준으로 판단하도록 만든다. 새 기능이나 원고가 “그럴듯해 보이는가”가 아니라 **독자가 처음부터 읽지 않아도 이해할 수 있는가, 실제 장비와 연결되는가, 수식의 변형을 납득할 수 있는가, 출처로 검증할 수 있는가**를 완료 기준으로 삼는다.

## 0. 이 매뉴얼을 사용하는 법

### 0.1 문서 우선순위

서로 다른 문서가 같은 사항을 다룰 때 다음 순서로 해석한다.

1. `AUTHORING_MANUAL.md`: 사용자 요구, 편집 범위, 콘텐츠·근거·완료 기준
2. `design-system.qmd`: 실제 렌더링되는 시각 구성 요소와 접근성 규칙
3. `platform/templates/concept-template.qmd`: 새 장의 기본 골격
4. `platform/glossary/glossary.yml`: 한글명·영문명·약어·정의의 단일 원본
5. `assets/README.md`: 스타일·스크립트·아이콘 자산의 책임과 변경 규칙
6. `ui-ux-research.qmd`: 디자인 결정의 외부 표준과 조사 근거

이 매뉴얼이 새 시각 블록을 요구하더라도 `design-system.qmd`에 렌더링 표본이 없으면 원고에서 임의 클래스를 양산하지 않는다. 먼저 웹·EPUB용 스타일과 정적 대체 표현을 구현하고 디자인 시스템에 표본을 추가한 다음 사용한다.

### 0.2 단일 원본 원칙

- 용어는 `platform/glossary/glossary.yml`에서 고친다. 생성된 용어집은 직접 편집하지 않는다.
- 색·간격·글꼴·모션 값은 `assets/styles/_tokens.scss`에서 관리한다. 장마다 숫자를 직접 넣지 않는다.
- 같은 교육적 의미에는 같은 의미 블록(semantic block)을 사용한다.
- 공통 설명을 여러 장에 복사하지 않는다. 개념 링크와 선수지식 체크리스트로 연결한다.
- 제품 사양과 가격은 원고가 아니라 “출처 묶음(source dossier)”에서 먼저 검증한 뒤 인용한다.

### 0.3 이번 작업에서 바꾸지 않는 것

- 엄밀한 수학 교재처럼 모든 정리를 처음부터 공리적으로 전개하는 것을 목표로 하지 않는다.
- 수치실험이나 그림을 증명으로 취급하지 않는다.
- 웹 상호작용 때문에 PDF·EPUB의 정보가 빠지게 만들지 않는다.
- 확인되지 않은 회사 내부 알고리즘을 사실처럼 단정하지 않는다.
- 미래 확장 분야의 상세 본문을 이번 세 장보다 먼저 쓰지 않는다.

### 0.4 2026-07-16 저장소 감사 기준선

이 절은 요구사항과 현재 구현을 혼동하지 않기 위한 기준선이다. 파일이 존재한다는 사실은 완료를 뜻하지 않는다.

- 세 대상 원고는 현재 모두 존재한다: `content/concepts/robotics/jacobian.qmd`, `content/concepts/statistics/maximum-likelihood-estimation.qmd`, `content/concepts/estimation/kalman-filter.qmd`.
- 개념 스키마와 웹 도구막대는 `difficulty`, `importance`, `importance_note`, `practice_frequency`, `practice_frequency_note`, `application_areas`를 지원한다.
- 웹 스크립트에는 읽기 진행 막대, 위치 슬라이더, 마지막 읽은 위치, 페이지 즐겨찾기, 절 책갈피의 로컬 저장 기반이 있다. 세 장에서 실제 접근성·복귀 동작을 검증하기 전에는 완료로 보지 않는다.
- 기존 학습 블록에 더해 자코비안 원고 작업 중 `concept-scorecard`, `block-metrics`, `derivation-step`, `simulation-placeholder`, `case-card`, `reference-note`, `algorithm-card`, `tradeoff-card` 계열의 웹·EPUB 스타일이 추가되었다.
- MLE와 칼만 필터 원고는 존재하지만 새 한국어·사례·지표·시각화 기준에 대한 재감사가 필요하다.
- PDF·EPUB 출력은 별도 빌드와 리더 검증이 필요하며 웹 기능의 존재를 정적 출력의 증거로 대신할 수 없다.

## 1. 사용자의 질문, 불편, 원하는 결과

| 본문에 영어 표현이 너무 많다. | 중간 진입 독자가 용어를 해독하느라 개념 흐름을 놓친다. | 문장은 한국어 중심으로 쓰고 첫 등장에 영문과 약어를 괄호로 병기한다. | 세 장의 표제·절 제목·본문을 검색해 불필요한 영문 단독 표현이 없다. |

| 개념의 중요도와 실제 사용 분야를 알고 싶다. | 무엇을 먼저 깊게 공부해야 할지 판단하기 어렵다. | 장·주제·핵심 공식에 난이도, 중요도, 실무 사용 빈도, 적용 분야를 표시한다. | 지표의 숫자와 근거 문장이 함께 표시된다. |

| 평문·수식·표·그림 사이가 답답하다. | 서로 다른 정보 단위가 한 덩어리로 보인다. | 자간은 조금 늘리고 문단·수식·표·그림·카드 전후 간격은 더 크게 벌린다. | 390 px와 1280 px 화면 및 Calibre에서 시각 검수한다. |
| 그래프·표·다이어그램이 더 필요하다. | 수식만으로는 변화·공간·데이터 흐름을 상상하기 어렵다. | 설명 목적이 분명한 시각화와 같은 내용을 설명하는 대체 텍스트를 둔다. | 세 장마다 필수 시각화 목록을 충족한다. |
| 실제 로봇·센서·브랜드 사례가 필요하다. | 추상 예제가 현업 데이터와 연결되지 않는다. | 모델·센서·단위·주기·잡음·데이터 흐름·가격 시점을 출처와 함께 제시한다. | 제조사 문서와 접근일이 있는 사례 카드가 존재한다. |

### 1.1 독자에게 약속하는 경험

독자는 어느 장의 중간 절로 바로 들어와도 다음 질문에 빠르게 답할 수 있어야 한다.

1. 이것은 한 문장으로 무엇인가?
2. 로봇에서 왜 필요한가?
3. 지금 내 수준에서 어디까지만 읽어도 되는가?
4. 이 기호의 모양(shape), 단위, 물리적 뜻은 무엇인가?
5. 이 식은 어떤 가정에서 나왔고 다음 줄에서 무엇이 바뀌었는가?
6. 실제 제품에서는 어떤 값이 들어오는가?
7. 자주 실패하는 이유와 진단 방법은 무엇인가?
8. 더 깊이 보려면 어디로 가야 하는가?

## 2. 현재 범위와 확장 경계

### 2.1 이번에 완성할 세 장

| 장 | 현재 파일 | 이번 완료 상태 |
|---|---|---|

“완성”은 분량이 많다는 뜻이 아니다. 이 매뉴얼의 장 완료 체크리스트를 통과하고 웹·PDF·EPUB에서 검증되었다는 뜻이다.

### 2.2 이번에는 목차만 만드는 분야

- 로봇공학: 좌표계, 순기구학, 역기구학, 경로계획, 제어, 위치추정, 동시적 위치추정 및 지도작성(SLAM)
- 인공지능: 지도학습, 신경망, 최적화, 강화학습, 로봇학습
- 역학·기구학: 강체 운동, 질량·관성, 라그랑주 역학, 접촉·마찰, 다물체계
- 센서공학: 엔코더, 관성측정장치(IMU), 라이다(LiDAR), 카메라, 힘·토크 센서, 거리·접촉 센서
- 구동기와 모듈: 전동기, 감속기, 드라이버, 브레이크, 전원, 통신 버스, 제어 주기

이 항목은 예정 개념(planned node), 관계, 짧은 한 문장 설명까지만 허용한다. 제품 목록이나 상세 이론을 먼저 집필해 세 장의 완성을 늦추지 않는다.

## 3. 설명의 깊이와 비순차 학습

### 3.1 결론 우선, 점진적 엄밀화

각 장은 다음 네 깊이를 명시적으로 구분한다.

| 깊이 | 독자의 질문 | 필요한 내용 | 생략해도 되는 것 |
|---|---|---|---|
| 직관(intuition) | “무슨 느낌이며 왜 쓰나?” | 30초 핵심, 물리적 이미지, 최소 용어, 실제 장면 | 증명, 일반 차원의 세부 계산 |
| 적용(application) | “값을 넣어 계산할 수 있나?” | 공식, 단위·모양, 풀이 예제, 알고리즘, 짧은 코드 | 정리의 완전한 증명 |
| 유도(derivation) | “왜 이 공식이 나오나?” | 출발점, 한 단계씩 변형, 사용 법칙, 가정, 검산 | 기초 공리부터의 전개 |
| 엄밀함·증명(proof) | “정확히 언제 성립하나?” | 정의역, 조건, 예외, 반례, 필요한 증명 | 연구 논문 수준의 주변 정리 전부 |

첫 화면에서 “이 장을 전부 읽어야 다음 장으로 갈 수 있다”는 인상을 주지 않는다. 깊이별 선수지식에는 반드시 `필수`, `도움`, `필요 없음`을 구분한다.

### 3.2 선수지식 체크리스트

각 장은 독자가 스스로 결정할 수 있는 진단 질문을 제공한다.

```text
[ ] 필수: 벡터를 좌표 열로 쓰고 행렬-벡터 곱의 모양을 계산할 수 있다.
[ ] 도움: 편미분의 의미를 설명할 수 있다.
[ ] 지금은 필요 없음: 측도론적 확률론의 엄밀한 구성.
```

선수 개념 이름만 나열하지 않는다. “왜 필요한지”, “어디까지 할 수 있어야 하는지”, “한 문장 진단 질문”을 함께 쓴다.

### 3.3 중간 진입 복구 장치

- 절이 시작될 때 새 약어가 나오면 그 절에서 다시 확장한다.
- 앞 절의 중요한 기호는 2~4줄짜리 “기호 다시 보기”로 복구한다.
- 관련 개념 링크에는 “먼저 읽기”, “막힐 때 보기”, “더 깊이 보기”라는 목적을 붙인다.
- 핵심 가정은 링크 뒤로 숨기지 않고 현재 페이지에도 요약한다.

## 4. 한국어 중심 용어·문체 규칙

### 4.1 기본 표기

첫 등장은 **한글 용어(English term, ABBR)** 순서로 쓴다.

- 가중치(weight)
- 엔코더(encoder)
- 편향(bias)
- 이상치(outlier)
- 자코비안(Jacobian)
- 최대우도추정(Maximum Likelihood Estimation, MLE)
- 칼만 필터(Kalman filter, KF)
- 관성측정장치(Inertial Measurement Unit, IMU)

약어가 없는 경우 약어 자리를 만들지 않는다. 두 번째부터는 한국어 또는 이미 확장한 약어를 사용한다. 영문을 매 문장 반복해 가독성을 떨어뜨리지 않는다.

### 4.2 다시 병기하는 경우

다음 경우에는 같은 페이지라도 영문이나 약어를 다시 병기한다.

- 새로운 큰 절이 독립적인 검색 진입점일 때
- 표·그림·알고리즘만 캡처해도 뜻이 통해야 할 때
- 번역이 여러 개라 원어가 식별자 역할을 할 때
- 약어를 마지막으로 쓴 뒤 긴 분량이 지났을 때
- 다른 분야에서 같은 한글 용어를 다르게 쓰는 경우

### 4.3 한국어로 바꿀 것과 유지할 것

| 원문 | 본문 권장 표현 | 비고 |
|---|---|---|
| weight | 가중치(weight) | 첫 등장 후 “가중치” |
| bias | 편향(bias) | 센서의 영점 편향과 통계적 편향을 구분 |
| outlier | 이상치(outlier) | “특이값”과 혼용하지 않음 |
| residual | 잔차(residual) | 오차(error)와 정의를 구분 |
| noise | 잡음(noise) | 확률모형인지 단순 교란인지 명시 |
| state | 상태(state) | 상태공간 문맥에서 첫 등장 병기 |
| measurement | 측정값(measurement) | 센서 원시값(raw data)과 구분 |
| update | 갱신(update) | 칼만 필터에서는 “측정 갱신” 권장 |
| prediction | 예측(prediction) | “시간 갱신”과 동의어 여부 설명 |
| likelihood | 우도(likelihood) | 확률과 혼동하지 않도록 정의 반복 |
| score | 점수함수(score function) | 일반 점수와 구분 |
| Hessian | 헤시안(Hessian) | 인명 유래임을 역사 블록에서 설명 가능 |
| actuator | 구동기(actuator) | 장비 문맥에서는 제품 분류도 병기 |
| smoothing | 평활화(smoothing) | 필터링(filtering)과 시간 방향 차이 설명 |
| localization | 위치추정(localization) | 위치·자세 추정을 포함하는지 명시 |

제품명, 회사명, 프로토콜, 함수·클래스·파일명은 공식 표기를 유지한다. 예: `NumPy`, `ROS 2`, `sensor_msgs/Imu`, `Universal Robots`. 다만 문장 구조와 설명은 한국어로 쓴다.

### 4.4 제목과 문장

- 장 제목은 한국어를 앞에 둔다: `칼만 필터(KF): 예측과 측정을 불확실성으로 결합하기`.
- 절 제목에 영문만 단독으로 쓰지 않는다: `Prediction` 대신 `예측(prediction)`.
- “~에 대해 알아보자”보다 이 절의 결론을 드러낸다.
- 한 문장에는 가능한 한 하나의 핵심 주장만 둔다.
- 명사 나열보다 주어와 동사가 있는 문장을 쓴다.
- `shape`, `rank`, `solver`, `tuning` 같은 단어를 습관적으로 단독 사용하지 않는다. 각각 `모양(shape)`, `계수(rank)`, `수치 해법(solver)`, `조정(tuning)`처럼 처음 병기한다.
- `algorithm`, `pseudo code`는 `알고리즘(algorithm)`, `의사코드(pseudocode)`로 쓴다.

### 4.5 용어 등록 절차

1. 중앙 용어집에 같은 개념이 있는지 검색한다.
2. 대표 한글명, 영문명, 약어, 독립적으로 읽히는 짧은 정의를 정한다.
3. 동의어와 피해야 할 오역을 기록한다.
4. `platform/glossary/glossary.yml`을 수정한다.
5. 본문의 첫 등장에 `.atlas-term`을 사용한다.
6. `make validate`로 중복·형식·생성 결과를 검사한다.

본문 정의와 중앙 용어집 정의가 다르면 중앙 용어집을 먼저 바로잡고 재생성한다.

## 5. 장의 표준 정보 구조

각 장은 아래 순서를 기본으로 한다. 필요하면 절을 더할 수 있지만, 독자가 답을 늦게 만나도록 순서를 뒤집지 않는다.

1. **30초 핵심**: 한 문장 정의, 로봇에서의 역할, 가장 중요한 한계
2. **학습 지표**: 난이도, 중요도, 실무 사용 빈도, 예상 시간, 적용 분야
3. **읽기 선택지**: 5~12분 빠른 읽기, 핵심 읽기, 유도·증명까지 읽기
4. **선수지식 체크리스트**: 깊이별 필수·도움·필요 없음
5. **이 장의 질문**: 독자가 끝에서 답할 3~5개 질문
6. **먼저 떠올릴 장면**: 물리·공학적 이미지와 한 장의 그림
7. **용어와 기호**: 한국어 중심 정의, 모양, 단위, 좌표계
8. **핵심 공식**: 역할, 입력·출력, 중요도·난이도, 성립 조건
9. **가장 작은 계산 예제**: 손으로 확인 가능한 수치
10. **단계별 유도**: 한 단계 한 변화, 이유와 근거, 검산
11. **알고리즘과 의사코드**: 입력·출력·가정·계산 순서
12. **짧은 코드**: 설명 없이 실행 가능한 최소 예제
13. **실제 시스템 사례**: 제품·센서·데이터 흐름·계산·결과
14. **시각화**: 그래프·표·공간 그림·데이터 흐름도
15. **실무 노트**: 흔한 실수, 실패·성공 사례, 장단점, 진단, 복구
16. **엄밀히 말하면**: 숨은 가정, 예외, 반례, 선택 증명
17. **숨 고르기**: 어원·역사·인물·인터뷰·뉴스 중 하나
18. **회상 점검**: 말로 설명, 계산, 실패 진단, 연결 문제
19. **관련 개념과 다음 경로**: 목적이 표시된 링크
20. **장 참고문헌과 자료**: 논문·표준·제품 문서·뉴스를 분류
21. **페이지 용어집**: 현재 장에서 실제 사용한 용어

### 5.1 장 분량보다 리듬을 관리한다

장문 단락을 세 개 이상 연속시키지 않는다. 다만 내용과 무관한 장식용 카드를 억지로 끼워 넣지 않는다. 긴 유도 뒤에는 다음 중 하나로 호흡을 바꾼다.

- 작은 수치 검산
- 그림 또는 변화 그래프
- 실제 센서 값
- 흔한 실수와 진단
- 출처가 있는 역사·인물 이야기
- 한 문장 회상 질문

## 6. 난이도·중요도·사용 빈도 지표

숫자는 장식이 아니라 독자의 선택 도구다. 숫자 옆에 반드시 한 문장 근거를 둔다.

### 6.1 난이도(difficulty) 1~5

| 값 | 기준 |
|---:|---|
| 1 | 용어와 그림만으로 핵심 직관을 이해할 수 있음 |
| 2 | 고등학교 대수와 간단한 벡터 계산이 필요함 |
| 3 | 학부 선형대수·미적분 또는 기초 확률 계산이 필요함 |
| 4 | 여러 선수 개념을 연결하고 행렬 유도를 따라야 함 |
| 5 | 엄밀한 증명, 수치 안정성, 비선형·확률 과정까지 함께 다룸 |

장 하나에 단일 난이도만 표시하지 않는다. 장 전체 난이도와 `직관`, `계산`, `유도`, `증명` 깊이별 난이도를 함께 쓸 수 있다.

### 6.2 중요도(importance) 1~5

| 값 | 기준 |
|---:|---|
| 1 | 역사·교양 또는 매우 특수한 확장 |
| 2 | 특정 하위 분야에서 선택적으로 사용 |
| 3 | 여러 응용을 이해하는 데 유용하지만 우회 가능 |
| 4 | 로봇공학의 주요 작업에서 반복 사용 |
| 5 | 여러 분야를 잇는 기반 개념이며 이후 장의 언어가 됨 |

중요도는 “수학적으로 아름답다”가 아니라 이 교재의 목표인 로봇공학·센서·제어·인공지능 학습에서의 연결성을 기준으로 한다.

### 6.3 실무 사용 빈도(practice frequency) 1~5

| 값 | 기준 |
|---:|---|
| 1 | 연구·특수 분석에서 드물게 직접 사용 |
| 2 | 특정 제품 또는 문제군에서 사용 |
| 3 | 분야에 따라 정기적으로 사용 |
| 4 | 일반적인 개발·분석 작업에서 자주 등장 |
| 5 | 구현, 로그 분석, 문서, 논문에서 거의 일상적으로 등장 |

빈도는 공개 코드·공식 문서·논문·채용 요구 등 확인 가능한 근거로 평가한다. 개인적 인상만으로 5점을 주지 않는다.

### 6.4 적용 단위

- 장: 첫 화면 지표 띠(metric strip)
- 주요 주제: 절 도입의 작은 칩
- 핵심 공식: 수식 카드 안의 난이도·중요도
- 실제 사례: 산업 분야, 성숙도, 근거 수준

현재 스키마가 지원하지 않는 새 메타데이터는 먼저 스키마·컴파일러·웹·EPUB 정적 표현까지 구현한 뒤 사용한다. 값이 원고에만 있고 독자 화면에 나오지 않는 상태를 완료로 보지 않는다.

## 7. 의미 블록과 재미 요소

### 7.1 현재 재사용하는 블록

| 클래스 | 한글 제목 | 사용 목적 |
|---|---|---|
| `.key-takeaway` | 핵심 결론 | 반드시 기억할 한두 문장 |
| `.term-card` | 용어 | 뜻, 어원, 번역 주의, 관련 용어 |
| `.formula-card` | 핵심 식 | 수식, 기호, 단위, 모양, 조건 |
| `.engineering-meaning` | 공학적 의미 | 수식과 실제 장치·신호 연결 |
| `.assumption-box` | 가정 | 성립 조건과 숨은 전제 |
| `.mistake-card` | 흔한 실수 | 잘못된 사용, 증상, 교정 |
| `.failure-mode` | 실패 양상 | 재현 조건, 관측 징후, 진단·복구 |
| `.history-card` | 숨 고르기 | 역사, 어원, 인물, 검증된 일화 |
| `.annotation-note` | 저자 노트·자료 주석 | 링크, 보충 정보, 편집상 설명 |
| `.learning-objectives` | 학습 목표 | 관찰 가능한 학습 행동 |

### 7.2 확장 블록 계약

현재 구현된 확장 블록은 아래 이름을 단일 표기로 사용한다. 같은 뜻의 새 이름을 만들지 않는다.

- `concept-scorecard`, `block-metrics`, `metric-badge`: 장·공식의 난이도·중요도·분야·시간
- `case-card`: 실제 제품과 데이터 흐름을 담는 사례. 주장 확실성과 확인일도 카드 안에 쓴다.
- `derivation-step`: 이전 식, 변경 부분, 근거, 결과
- `simulation-placeholder`: 현재 정적 결과와 미래 상호작용 계약
- `reference-note`: 본문 흐름을 끊지 않는 짧은 자료 링크
- `algorithm-card`: 입력·출력·가정·계산 순서를 보이는 알고리즘
- `tradeoff-card`: 장점·단점·선택 기준·실무 노하우
- `origin-card`, `interview-card`, `news-card`, `fun-fact-card`: 어원·인터뷰·뉴스·재미있는 사실

웹·EPUB 스타일이 존재해도 디자인 시스템 표본, 좁은 화면, 키보드, 인쇄 출력에서 검증되어야 사용할 수 있다. 색만으로 블록의 의미를 구분하지 않는다. 아이콘, 항상 보이는 한글 제목, 테두리를 함께 둔다. EPUB에서 배경색이 사라져도 제목과 문서 순서만으로 의미가 남아야 한다.

### 7.3 역사·어원·인터뷰·뉴스 규칙

- 본문 핵심 주장을 대신하지 않고 1~4분 안에 읽는 길이로 쓴다.
- “최초”, “발명”, “유래”는 1차 자료 또는 신뢰할 수 있는 역사 자료로 확인한다.
- 유명한 일화가 사실인지 불분명하면 “널리 전해지는 이야기이나 확인되지 않았다”고 표시한다.
- 인터뷰 발언은 발언자, 매체, 날짜, 원문 링크를 쓴다.
- 뉴스는 기술 사양의 권위 자료가 아니라 시대적 맥락 자료로 취급한다.
- 노벨상, 회사 사건, 제품 성공담은 개념과 직접 연결되는 경우에만 넣는다.

## 8. 수식·유도·증명 편집 규칙

### 8.1 핵심 수식 카드의 필수 항목

1. 이 수식이 답하는 질문
2. 수식 자체
3. 기호별 한글 이름과 의미
4. 벡터·행렬의 모양(shape)
5. 물리 단위와 좌표계
6. 성립 가정
7. 중요도와 난이도
8. 가장 작은 수치 예제
9. 실패하거나 정의되지 않는 조건
10. 다음 수식과의 연결

### 8.2 한 단계에 한 변화

수식 유도는 다음 형식의 “변형 장부”로 작성한다.

| 단계 | 식 | 바뀐 부분 | 근거 | 조건 |
|---:|---|---|---|---|
| 0 | 출발 식 | 없음 | 정의 또는 앞 절 결과 | 적용 범위 |
| 1 | 다음 식 | 실제로 바뀐 항 | 분배법칙, 연쇄법칙, 로그의 단조성 등 | 법칙의 조건 |
| 2 | 다음 식 | 소거·묶기·치환 | 사용한 등식 또는 가정 | 0이 아님, 가역성 등 |
| 3 | 결과 식 | 최종 정리 | 최적 조건 또는 정의 | 유일성·존재 조건 |

식 하나에서 두 가지 조작을 동시에 하지 않는다. 불가피하면 중간 줄을 추가한다.

### 8.3 시각적 표시는 의미와 함께 쓴다

- 색은 “새로 바뀐 항”, “고정된 항”, “소거되는 항”처럼 일관된 의미로만 쓴다.
- 색만으로 의미를 전달하지 않고 밑괄호, 굵기, 라벨, 단계 번호를 함께 쓴다.
- 벡터 굵기는 수학적 종류를 나타낼 때만 사용한다. 강조 목적으로 기호를 임의로 굵게 만들지 않는다.
- 긴 식은 `aligned` 또는 변형 장부로 의미 단위 개행한다.
- 한국어 설명을 긴 TeX `\text{}` 안에 넣지 않는다. 수식 밖의 열이나 주석으로 둔다.
- `=`, `\iff`, `\Rightarrow`, `\approx`, `\propto`를 뜻에 맞게 구분한다.
- 등호를 쓸 수 없는 근사·선형화 단계를 등호로 연결하지 않는다.

### 8.4 매 유도의 검산

유도 끝에는 가능한 항목을 최소 두 개 확인한다.

- 차원·모양 검사
- 물리 단위 검사
- 극한 또는 특수값 검사
- 부호와 양의 정부호성 검사
- 작은 숫자를 넣은 직접 계산
- 유한차분 또는 자동미분과 비교
- 모의 데이터에서 원래 목적함수와 비교

### 8.5 “엄밀히 말하면”의 범위

엄밀한 절은 이해를 방해하는 장벽이 아니라 앞의 직관을 정확히 좁히는 장치다.

- 정의역·공역, 연속·미분 가능성, 가역성, 계수 조건을 명시한다.
- 확률모형에서는 독립성, 동일분포, 정규성, 잡음의 평균·공분산을 구분한다.
- 정리가 어떤 조건 없이도 성립하는 부분과 특정 가정이 필요한 부분을 나눈다.
- 반례 하나를 제공해 가정의 역할을 보인다.
- 긴 주변 정리는 별도 링크로 보내되 현재 결론에 필요한 핵심은 남긴다.

## 9. 알고리즘·의사코드·코드

### 9.1 알고리즘 블록

알고리즘에는 다음이 보여야 한다.

- 이름과 목적
- 입력, 출력
- 기호의 모양과 단위
- 가정과 초기화
- 번호가 있는 계산 단계
- 반복 조건과 종료 조건
- 계산량 또는 병목
- 실패 조건과 안전한 수치 해법

의사코드는 특정 프로그래밍 언어 문법을 흉내 내기보다 논문과 구현을 연결한다. `inverse(A)` 대신 선형계 풀이가 적절하면 그 차이를 명시한다.

### 9.2 코드 규칙

- 코드는 짧고 실행 가능하게 작성한다.
- 사용자의 요구에 따라 코드 바로 뒤에 줄별 해설을 붙이지 않는다.
- 대신 코드 앞에 입력과 기대 결과를 1~3문장으로 쓴다.
- 난수는 씨앗(seed)을 고정한다.
- 단위, 표본 주기, 배열 모양을 변수명 또는 짧은 주석으로 남긴다.
- 핵심 함수는 `courseware/src/`로 분리해 테스트할 수 있게 한다.
- 그림을 생성하는 코드는 원자료와 출력 파일의 관계를 재현할 수 있어야 한다.
- 장식용 외부 라이브러리를 늘리지 않는다. 기본은 Python, NumPy, SciPy, Matplotlib, 필요 시 pandas·Plotly를 사용한다.

## 10. 시각화 설계

시각화는 “있으면 멋진 그림”이 아니라 문장보다 관계를 빨리 이해하게 할 때만 사용한다.

### 10.1 무엇을 어떤 시각화로 보여줄지

| 설명 목적 | 우선 시각화 |
|---|---|
| 변수 사이 정확한 대응 | 표 |
| 계산 순서와 데이터 흐름 | 흐름도 |
| 시간에 따른 상태·측정·추정 변화 | 시계열 그래프 |
| 2차원 불확실성 | 공분산 타원 |
| 좌표 변환·속도 관계 | 좌표축과 화살표 도식 |
| 행렬의 국소 민감도 | 입력 격자와 출력 변형 |
| 특이점과 조건수 | 자세 변화 애니메이션의 정적 연속 프레임 또는 곡선 |
| 우도와 최적점 | 모수-우도 곡선 또는 등고선 |
| 알고리즘 비교 | 같은 축·같은 데이터의 겹친 그래프와 요약표 |

### 10.2 모든 그림의 필수 요소

- 결론을 말하는 한글 제목
- 축 이름, 단위, 범례
- 본문에서 그림을 해석하는 문장
- 대체 텍스트와 필요 시 긴 설명
- 데이터·코드·제품 사진의 출처와 사용 허가
- 색각 이상과 흑백 출력에서도 구분되는 선·표식·라벨
- 캡션에 “무엇을 봐야 하는지” 한 문장

이미지 안에 작은 글씨로 핵심 설명을 박아 넣지 않는다. 확대·리플로우가 가능한 본문과 캡션에 남긴다.

### 10.3 세 장의 최소 시각화

**자코비안**

- 입력 공간의 작은 변화가 출력 공간의 화살표로 옮겨지는 그림
- 2자유도 평면 로봇 팔의 관절속도와 말단속도
- 열벡터별 관절 기여도
- 특이점 전·근처·정확한 특이점의 세 자세
- 특잇값과 조작성(manipulability)의 변화 그래프
- 감쇠 최소제곱 전후 속도 폭주 비교

**최대우도추정**

- 확률과 우도의 “고정하는 대상” 비교
- 베르누이 우도와 로그우도 곡선
- 정규 잡음에서 잔차 제곱합과 우도의 대응
- 엔코더 보정 데이터, 적합선, 잔차 그래프
- 이상치가 MLE에 미치는 영향
- 헤시안 곡률과 추정 불확실성의 관계

**칼만 필터**

- 예측 타원·측정 띠·갱신 타원의 기하
- 참값·잡음 측정·예측·추정 시계열
- 예측–혁신–보정 데이터 흐름도
- 칼만 이득과 측정 잡음의 관계
- 적절한 조정과 잘못된 `Q`, `R`의 잔차 비교
- 결측·이상치·지연 측정 실패 사례

## 11. 정적 시뮬레이션 자리표시자

이번 단계에서는 문서와 정적 결과를 먼저 완성한다. 자리표시자는 다음 내용을 포함한다.

1. 실험 질문
2. 미래 조작값: 슬라이더·선택 단추·재생 단추의 이름과 범위
3. 고정된 기본 입력값과 단위
4. 현재 보이는 정적 그래프 또는 표
5. 값을 바꾸면 무엇이 변할지 한 문장 예상
6. 미래 계산 함수의 입력·출력 자료 구조
7. `상호작용 기능 준비 중 — 현재는 기준값의 정적 결과`라는 명시적 상태

작동하지 않는 단추를 작동하는 것처럼 보이게 만들지 않는다. 비활성 상태는 키보드와 화면 읽기 프로그램에도 전달한다. PDF·EPUB에서는 조작 UI를 제거하고 기준값·결과·QR 또는 웹 링크만 남긴다.

### 11.1 세 장의 미래 실험 계약

- 자코비안: 관절각 슬라이더 → 로봇 자세, 자코비안 열, 특잇값, 말단속도
- 최대우도추정: 표본 수·잡음·이상치 비율 슬라이더 → 우도 곡선, MLE, 잔차
- 칼만 필터: `Q`, `R`, 결측률 슬라이더 → 칼만 이득, 추정 시계열, 일관성 지표

## 12. 실제 제품·브랜드·사례와 근거 정책

### 12.1 실제 사례 카드의 필수 필드

| 필드 | 기록 내용 |
|---|---|
| 시스템 | 제조사, 제품명, 세대·선택 사양, 확인 날짜 |
| 작업 | 로봇이 실제로 수행하는 일 |
| 센서 | 제조사·모델, 측정 원리, 장착 위치·좌표계 |
| 주요 사양 | 측정 범위, 해상도, 정확도, 표본률, 지연, 단위 중 관련 항목 |
| 연결 | 버스·프로토콜·메시지 형식·시간표시 방식 중 확인된 항목 |
| 원시 데이터 | 실제로 들어오는 필드와 예시 숫자 |
| 전처리 | 보정, 좌표변환, 시간 동기화, 이상치 제거 |
| 알고리즘 | 자코비안·MLE·칼만 필터의 어느 단계에서 어떻게 쓰는지 |
| 결과 | 계산된 값, 해석, 기대되는 개선과 한계 |
| 가격 | 통화, 지역, 세금·배송 포함 여부, 정가·판매가 구분, 확인일 |
| 출처 | 제조사 자료, 데이터시트, 매뉴얼, 논문, 코드, 보조 기사 |
| 확실성 | 확인된 사실, 합리적 추론, 교육용 가정 중 하나 |

모든 사양을 베끼지 않는다. 현재 개념과 계산에 쓰는 사양만 선택한다.

### 12.2 출처 우선순위

1. 국제 표준, 제조사 데이터시트·매뉴얼·개발자 문서, 공개 규제 문서
2. 원 논문, 학술대회 논문, 공식 기술 보고서
3. 제조사 기술 블로그·공개 코드·특허
4. 신뢰할 수 있는 인터뷰와 전문 매체
5. 유통사·판매점·커뮤니티 자료

가격은 제조사 또는 공식 유통사의 날짜가 있는 자료를 우선하며, 5번 자료를 쓸 때는 “가격 참고용”이라고 제한한다. 뉴스나 인터뷰만으로 정확도·안전성·내부 알고리즘을 단정하지 않는다.

### 12.3 시간에 따라 바뀌는 사실

- 가격·펌웨어·사양·제품 구성·회사 기술은 집필 시점에 다시 웹에서 확인한다.
- 모든 값에 확인일과 지역을 붙인다.
- 정확한 현재가를 유지할 수 없으면 가격대와 기준 시점을 쓰거나 가격을 생략한다.
- 단종 제품은 단종 여부와 마지막 확인 시점을 표시한다.
- 제품 이름이 같아도 선택 사양에 따라 센서 구성이 다르면 구성을 특정한다.

### 12.4 독점 기술과 과장 방지

예를 들어 “현재 Tesla의 자율주행에 칼만 필터가 핵심으로 들어간다”는 문장은 공개된 1차 자료가 직접 확인할 때만 사실로 쓴다. 공개 자료가 구조 일부만 보여 주면 다음처럼 구분한다.

- **확인된 사실:** 공식 문서가 명시한 내용
- **분석자의 추론:** 공개 구조로부터 합리적으로 추정하지만 회사가 확인하지 않은 내용
- **교육용 대응 예시:** 그 시스템과 비슷한 문제를 이 알고리즘으로 풀어 보는 예제

회사 홍보 표현을 성능의 독립적 검증처럼 옮기지 않는다. “완벽”, “혁신적”, “업계 최고” 같은 표현은 비교 조건과 근거 없이는 사용하지 않는다.

### 12.5 실제 사례 후보를 고르는 기준

이번 세 장은 공개 문서·데이터·사양이 충분한 플랫폼을 우선한다. 후보로 산업용 협동로봇, 공개 개발자 문서가 있는 이동로봇, 상용 라이다·관성측정장치·엔코더를 검토할 수 있다. **후보 제품명을 먼저 정답처럼 원고에 넣지 않고**, 다음 자료가 모인 뒤 채택한다.

- 공식 데이터시트와 사용자 매뉴얼
- 데이터 필드 또는 통신 인터페이스 설명
- 계산에 필요한 단위·표본률·오차 사양
- 사용 가능한 사진·도식 또는 직접 제작 가능한 개략도
- 날짜가 있는 가격 근거 또는 가격 생략 결정
- 알고리즘과 제품을 잇는 공개 근거

### 12.6 사례 풀이의 데이터 흐름

각 사례는 다음 화살표를 실제 숫자와 함께 완주한다.

```text
물리 현상
→ 센서 변환
→ 원시 숫자와 단위
→ 시간표시·좌표계
→ 보정·이상치 처리
→ 자코비안 / MLE / 칼만 필터 입력
→ 중간 계산
→ 출력과 불확실성
→ 제어·판단에서의 사용
```

“센서 융합에 사용한다”에서 멈추지 않는다. 어느 센서의 어떤 값이 상태벡터의 어느 항과 측정행렬의 어느 행에 들어가는지 보여 준다.

## 13. 실무 실수·실패·성공·장단점

각 장에는 최소 하나의 재현 가능한 실패 사례가 있어야 한다. 실패 카드의 순서는 다음과 같다.

1. 상황과 설정
2. 잘못된 판단 또는 구현
3. 로그·그래프에서 보이는 증상
4. 원인
5. 최소 재현 계산 또는 코드
6. 진단 순서
7. 수정 방법
8. 수정 후 확인 지표
9. 남는 한계

### 13.1 세 장에서 반드시 다룰 실패

**자코비안**

- 좌표계와 프레임을 섞음
- 선속도와 각속도의 단위를 섞음
- 특이점 근처에서 단순 역행렬 또는 의사역행렬로 속도가 폭주함
- 수치 자코비안의 유한차분 간격이 너무 크거나 작음
- 행·열 배치를 전치해 우연히 모양만 맞음

**최대우도추정**

- 확률과 우도를 같은 함수로 읽음
- 로그우도에서 부호를 바꾸고 최댓값·최솟값을 뒤집음
- 독립·동일분포 가정을 확인하지 않음
- 이상치가 많은데 정규 잡음 MLE를 그대로 적용함
- 분산 MLE의 표본 수 분모와 불편추정량을 혼동함
- 설계행렬 조건이 나쁜데 정규방정식만 사용함

**칼만 필터**

- `Q`, `R`, `P`의 단위와 표본시간을 무시함
- 공분산을 단순 튜닝 숫자로 취급함
- 시간 동기화와 좌표계를 무시함
- 측정 이상치와 결측을 정상 갱신에 넣음
- 명시적 역행렬로 칼만 이득을 계산함
- 단순 공분산 갱신의 수치 오차로 대칭성·양의 준정부호성을 잃음
- 비선형·비정규 문제에서 선형 칼만 필터의 보장을 과장함

성공 사례는 결과만 자랑하지 않는다. 성공 조건, 비교 기준선, 평가 지표, 실패했던 대안과 비용을 함께 쓴다.

## 14. 장별 집필 설계서

### 14.1 자코비안(Jacobian)

**핵심 약속:** 자코비안은 “여러 입력의 아주 작은 변화가 여러 출력의 순간 변화로 어떻게 옮겨지는지”를 나타내는 국소 선형 지도다.

필수 전개:

1. 1변수 미분에서 변화율을 회상한다.
2. 스칼라→벡터, 벡터→스칼라, 벡터→벡터 함수의 미분 모양을 비교한다.
3. 자코비안의 행·열 규약을 명시하고 다른 관례를 경고한다.
4. 전미분과 국소 선형화를 그림으로 설명한다.
5. 합성함수의 연쇄법칙을 자코비안 곱으로 유도한다.
6. 2자유도 평면 팔의 순기구학에서 자코비안을 한 항씩 미분한다.
7. 각 열을 “해당 관절 하나가 만드는 말단 속도”로 해석한다.
8. 기하 자코비안과 해석 자코비안을 구분한다.
9. 속도·힘/토크 관계와 전치 자코비안의 공학적 의미를 다룬다.
10. 계수, 영공간, 특이점, 조건수, 특잇값을 연결한다.
11. 역행렬, 의사역행렬, 감쇠 최소제곱의 사용 조건을 비교한다.
12. 유한차분·자동미분·해석 자코비안의 검산을 보여 준다.
13. 실제 로봇의 관절 상태 데이터에서 말단 속도를 계산하는 사례를 완주한다.

필수 알고리즘:

- 해석 자코비안 계산
- 중심 유한차분 자코비안
- 특이점 감시와 감쇠 최소제곱 속도 명령

### 14.2 최대우도추정(MLE)

**핵심 약속:** 관측 데이터는 고정하고, 그 데이터를 가장 그럴듯하게 만드는 모수를 찾는다.

기존 장에서 유지·강화할 것:

1. 확률과 우도에서 무엇을 고정하는지 그림과 표로 구분한다.
2. 곱 형태 우도를 로그우도로 바꾸는 이유를 단조성·수치 안정성과 함께 설명한다.
3. 베르누이·정규분포의 MLE를 중간 단계 없이 유도한다.
4. 점수함수, 헤시안, 관측 정보, 피셔 정보의 역할을 한국어 중심으로 다시 쓴다.
5. 정규 잡음 가정 아래 MLE와 최소제곱의 동치를 가정별로 설명한다.
6. 가중 최소제곱과 센서별 분산을 연결한다.
7. 엔코더 척도·편향 보정 사례에 실제 장비 사양·단위·데이터 흐름을 추가한다.
8. 이상치와 잡음모형 불일치 실패를 시각화한다.
9. 유일성·식별가능성·경계해·퇴화 사례를 명시한다.
10. 짧은 Python 코드와 논문형 알고리즘을 함께 둔다.

필수 알고리즘:

- 일반 로그우도 최대화 절차
- 선형 엔코더 척도·편향 MLE
- 헤시안 또는 정보행렬을 이용한 국소 불확실성 근사

### 14.3 칼만 필터(KF)

**핵심 약속:** 동역학의 예측과 센서 측정을 각각의 불확실성에 따라 반복적으로 절충한다.

필수 전개:

1. 1차원 두 추정값의 분산 가중 결합에서 시작한다.
2. 상태, 제어입력, 동역학, 측정, 과정잡음, 측정잡음을 실제 단위로 정의한다.
3. 예측 평균과 공분산을 한 단계씩 유도한다.
4. 혁신(innovation), 혁신 공분산, 칼만 이득, 상태 갱신을 순서대로 설명한다.
5. 공분산 갱신과 조지프 형식(Joseph form)의 수치적 의미를 다룬다.
6. 가우스 조건부 분포, 가중 최소제곱, 직교성 원리와의 연결을 깊이별로 나눈다.
7. 실제 이동로봇의 엔코더·관성측정장치·위치 센서 데이터를 시간·좌표계와 함께 구성한다.
8. 예측만 수행하는 시점과 측정이 도착하는 시점을 구분한다.
9. 이상치 문턱값, 결측, 지연, 서로 다른 표본률을 다룬다.
10. `Q`, `R`, `P0`의 물리 단위와 추정 방법을 설명한다.
11. 잔차 일관성 검사를 그래프로 보여 준다.
12. 선형 칼만 필터, 확장 칼만 필터(EKF), 평활화의 경계를 짧게 안내한다.

필수 알고리즘:

- 이산 선형 칼만 필터 예측–갱신
- 이상치·결측 측정이 있는 안전한 갱신
- 비동기 다중 센서 사건 순서 처리

## 16. 웹·PDF·EPUB 정보 경험

### 16.1 웹: 기준 배포판

웹은 위키·블로그형 비순차 학습의 기준 경험이다.

- 왼쪽 전체 목차와 현재 위치
- 오른쪽 장 내부 목차
- 페이지 상단 읽기 진행률
- 마지막 읽은 절을 로컬에 저장하고 복귀 링크 제공
- 장·절 즐겨찾기와 즐겨찾기 목록
- 검색, 관련 개념, 선수지식, 역링크(backlink)
- 30초 핵심으로 바로 가는 단축 링크
- 참고문헌·용어집·실제 사례로 바로 가는 장 내 링크
- 정적 시뮬레이션 자리표시자
- 향후 상호작용은 점진적 향상(progressive enhancement)으로 추가

로그인이나 서버가 없어도 기본 진도와 즐겨찾기가 로컬 저장소에서 동작하도록 설계한다. 저장 실패 시 본문 읽기가 막히면 안 된다.

### 16.2 PDF

- 실제 페이지 번호, 책갈피, 클릭 가능한 목차와 내부 링크를 제공한다.
- 웹 전용 단추·진행 막대는 인쇄하지 않는다.
- 색 없이도 블록 제목과 테두리로 역할이 구분되어야 한다.
- 넓은 표와 수식은 잘리지 않도록 별도 방향·축소·개행을 검토한다.

### 16.3 EPUB

EPUB의 “페이지 번호”와 “책갈피”는 화면 크기·글꼴·리더에 따라 달라지며 대체로 읽기 프로그램이 관리한다. 원고가 고정 페이지 슬라이더를 강제하지 않는다. 대신 다음을 보장한다.

- 계층이 올바른 목차와 장·절 링크
- 문서 랜드마크와 장 제목
- 리더의 책갈피·마지막 위치 기능이 사용할 안정적인 절 식별자
- 페이지별 용어집과 정적 참고문헌
- MathML 수식의 리플로우와 국소 넘침 처리
- 상호작용 없이도 모든 핵심 정보가 남는 정적 대체 표현

Calibre 한 곳에서만 맞으면 완료가 아니다. 최소한 Calibre에서 자세히 검사하고, EPUB 구조·링크·CSS·MathML도 자동 검사한다.

### 16.4 간격과 가독성 목표

- 현재 본문 자간은 크게 바꾸지 않고 한글 획 사이 여유를 조금 더 늘린다.
- 문단 사이 간격보다 절·수식·표·그림·카드 전후 간격을 더 크게 둔다.
- 표·그래프·그림은 앞 설명과 뒤 해석에서 각각 분리되어 보여야 한다.
- 좁은 화면에서 평문은 가로 스크롤 없이 읽히고 표·긴 수식만 국소 스크롤한다.
- 간격 값은 토큰으로 조정하고 장별 인라인 스타일을 금지한다.
- 확대, 사용자 글자 간격 재정의, 다크 모드에서 잘림·겹침이 없어야 한다.

시각 검수는 “숫자가 커졌다”가 아니라 긴 한영 혼합 문단, 연속 수식, 넓은 표, 캡션이 있는 그림을 실제 세 장에서 비교해 판단한다.

## 17. 작업 순서

### 17.1 0단계 — 요구 고정

- 이 매뉴얼을 최신 상태로 만든다.
- 기존 원고·메타데이터·디자인 시스템·출력 상태를 감사한다.
- 세 장 이외의 본문 집필을 동결한다.

### 17.2 1단계 — 공통 편집 기반

- 한국어 중심 용어 목록과 오역 방지 목록을 확정한다.
- 중요도·사용 빈도·분야 메타데이터의 스키마와 출력 표현을 구현한다.
- 새 의미 블록을 웹·EPUB·인쇄용으로 구현하고 디자인 시스템에 표본을 만든다.
- 문단·수식·표·그림 간격 토큰을 조정한다.
- 즐겨찾기·마지막 위치·장 내 바로가기의 정적·웹 계약을 구현한다.
- 정적 시뮬레이션 자리표시자를 만든다.

### 17.3 2단계 — 출처 묶음

세 장별로 수학 출처, 제품 후보, 데이터시트, 가격 근거, 실제 데이터 형식, 역사 자료를 조사한다. 제품과 회사에 관한 최신 주장은 조사 날짜에 웹에서 다시 확인한다.

### 17.4 3단계 — 자코비안 완성

새 장이므로 표준 구조와 새 편집 시스템을 처음부터 적용한다. 이 장에서 수식 변형 장부, 로봇 도식, 공식 지표의 대표 패턴을 확정한다.

### 17.5 4단계 — MLE 재편집

기존 수학 내용을 보존하면서 영어 단독 표현, 긴 유도, 제품 사례, 지표, 출처, 시각화를 새 기준에 맞춘다. 수학적 의미가 바뀌는 기계적 일괄 번역은 하지 않는다.

### 17.6 5단계 — 칼만 필터 확장

공통 패턴을 재사용하고 실제 다중 센서 데이터 흐름, 비동기 처리, 실패 진단을 집중적으로 완성한다.

### 17.7 6단계 — 연결과 확장 골격

- 세 장 사이의 링크를 목적별로 연결한다.
- 미래 로봇공학·인공지능·역학·기구학·센서공학 개념은 예정 노드와 한 문장 설명만 추가한다.
- 중복 정의를 중앙 용어집으로 통합한다.

### 17.8 7단계 — 검증과 독자 시나리오

- 자동 검사, 빌드, 브라우저, PDF, EPUB을 모두 검증한다.
- 각 장의 중간 절로 직접 들어오는 시나리오를 검사한다.
- 실제 제품 숫자가 출처와 계산에서 일치하는지 역추적한다.
- 수식의 각 줄이 전 줄에서 어떤 법칙으로 나왔는지 제3자가 설명할 수 있는지 확인한다.

- [ ] 짧은 코드는 실행 가능하고 난수를 재현할 수 있다.

### 18.4 사례와 근거

- [ ] 실제 사례가 **실제 원시값 또는 명시적으로 합성된 원시 형식 표본**에서 알고리즘 출력까지 완주하며, 실제 자료와 합성 자료의 경계를 명시한다.
- [ ] 제품의 모델·구성·사양·확인일이 특정된다.
- [ ] 가격을 썼다면 통화·지역·포함 항목·확인일이 있다.
- [ ] 확인된 사실, 추론, 교육용 가정을 구분한다.
- [ ] 실제 사용 주장을 공식 문서·논문·공개 코드로 검증한다.
- [ ] 실패 사례가 증상·원인·진단·수정·재검증을 포함한다.
- [ ] 장별 참고문헌이 자료 유형별로 정리된다.

### 18.5 시각·상호작용·출력

- [ ] 장별 최소 시각화 목록을 충족한다.
- [ ] 모든 그림에 제목·축·단위·캡션·대체 텍스트가 있다.
- [ ] 평문·수식·표·그래프·이미지의 간격이 시각적으로 구분된다.
- [ ] 웹 읽기 진행률, 마지막 위치, 즐겨찾기, 바로가기가 동작한다.
- [ ] 정적 시뮬레이션은 준비 중임을 명시하고 기준 결과를 보여 준다.
- [ ] JavaScript가 없어도 핵심 본문을 읽을 수 있다.
- [ ] 320~390 px에서 평문 가로 넘침이 없다.
- [ ] 라이트·다크·키보드·감소된 모션에서 의미 손실이 없다.
- [ ] PDF의 목차·페이지 번호·책갈피·내부 링크가 동작한다.
- [ ] EPUB의 목차·절 링크·MathML·정적 대체 표현이 동작한다.

### 18.6 범위 통제

- [ ] 이번 단계에서 **새로 상세 집필·편집하는 대표 장(flagship) 범위**는 자코비안·MLE·칼만 필터 세 장에 한정했다.
- [ ] 이미 존재하는 핵심(core) 콘텐츠와 사용자가 명시적으로 승인한 입자 필터(Particle Filter, PF)·마르코프 연쇄 몬테카를로(Markov Chain Monte Carlo, MCMC) 유도·증명 트랙은 보존하며, 이를 위 항목의 범위 위반으로 세지 않는다. 단, 보존이 새로운 상세 편집 권한을 자동으로 뜻하지는 않는다.
- [ ] 위의 대표 장·기존 핵심 콘텐츠·승인된 PF/MCMC 트랙 밖의 미래 분야는 예정 구조와 한 문장 설명을 넘지 않는다.
- [ ] 새 블록과 새 메타데이터가 디자인 시스템·스키마·출력에 먼저 반영되었다.
- [ ] 생성 파일을 직접 편집하지 않았다.

## 19. 검증 절차

자동 검사를 먼저 실행하되, 통과를 시각·내용 완성의 증거로 과장하지 않는다.

```bash
make validate
make test
make web
make book
make proof
```

추가 수동 검증:

1. 390 px 모바일과 1280 px 데스크톱에서 세 장을 처음부터 끝까지 본다.
2. 다크 모드와 200% 확대에서 수식·표·카드 겹침을 확인한다.
3. 키보드만으로 목차·용어·즐겨찾기·바로가기를 사용한다.
4. JavaScript를 끄고 핵심 정보와 링크가 남는지 본다.
5. PDF에서 목차, 페이지 번호, 책갈피, 내부·외부 링크를 확인한다.
6. Calibre에서 글꼴 크기·행간·테마를 바꾸며 EPUB을 확인한다.
7. 제품 사례의 숫자를 원출처에서 다시 찾아 계산에 대입한다.
8. 세 장에서 영문 단독 절 제목과 정의되지 않은 약어를 검색한다.
9. 수식 유도 한 줄을 무작위로 골라 전 줄에서의 근거를 추적한다.
10. 링크 검사와 EPUB 압축·목차·MathML 구조 검사를 실행한다.

## 20. 편집 결정 기록과 변경 관리

새 요구가 생기면 먼저 이 문서의 “사용자의 질문, 불편, 원하는 결과”와 완료 체크리스트를 갱신한다. 구현부터 시작해 요구의 이유를 잃지 않는다.

중요한 편집 결정은 다음 형식으로 기록한다.

```text
날짜:
대상 장/기능:
사용자 불편:
검토한 선택지:
선택:
선택 이유:
웹/PDF/EPUB 영향:
검증 방법:
남은 위험:
```

숫자 지표, 번역, 제품 사례, 수식 관례를 바꾸면 바뀐 이유와 영향받는 장을 기록한다. 한 장의 편의를 위해 공통 규칙을 조용히 깨지 않는다.

## 부록 A. 실제 사례 카드 저작 틀

```markdown
::: {.case-card}
### 실제 시스템 · [작업을 드러내는 제목]

**확실성:** 확인된 사실 / 분석자의 추론 / 교육용 가정  
**시스템:** 제조사, 제품, 선택 사양, 확인일  
**센서:** 제조사, 모델, 측정 원리, 장착 좌표계  
**데이터:** 필드, 모양, 단위, 표본률, 시간표시  
**가격:** 통화, 지역, 세금·배송·선택 사양, 확인일

물리 현상 → 원시 데이터 → 전처리 → 알고리즘 입력 → 중간 계산 → 출력 → 사용

#### 숫자로 따라가기

실제 원시값 또는 명시적으로 합성된 원시 형식 표본에서 시작해 단계별로 계산한다. 합성 표본은 실제 로그·제품 출력·성능 측정값이라고 부르지 않으며, 실제 자료와 같은 자료 묶음인 것처럼 이어 붙이지 않는다.

#### 실무에서 조심할 점

좌표계, 동기화, 잡음, 이상치, 지연, 수치 안정성을 확인한다.

**근거:** [공식 데이터시트](링크), [사용자 매뉴얼](링크), [적용 근거](링크)  
**자료 확인일:** YYYY-MM-DD
:::
```

`case-card`가 지원되지 않는 정적 출력에서도 같은 제목과 필드 순서가 남아야 한다.
