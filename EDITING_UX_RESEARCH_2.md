## 1. 한글(HWP) — 한국어 감독자의 UX 기대치 [전부 검증]

- **[검증]** 한/글의 '변경 내용 추적'은 Word와 동형의 관례를 완비한다: 삽입·삭제의 시각적 구분, 검토자별 색상과 검토자 단위 필터링, 여백 풍선(변경 안내문) + 본문-풍선 연결선 옵션. 한국어 감독자는 웹 교정 도구에서도 이 세 가지를 기본 기대치로 갖는다.
- **[검증]** 보기 모드는 '원본 / 원본+변경 / 최종본+변경 / 최종본' 4가지 — 변경 전후 토글 뷰가 표준 기대치다.
- **[검증]** 수락/거부는 8종 명령의 계층 구조다: 변경 1건 단위 적용/취소, **"적용(취소) 후 다음으로 이동"** 순차 내비게이션, '표시된 변경 모두'/'문서 전체' 일괄 처리. "변경 1건 = 검토·수락 1단위 + 순차 이동 + 계층적 일괄"이 감독자의 기대 모델이다.
- **[검증]** 한/글은 **'교정 부호'를 검토 탭의 1급 기능으로 제공하며, 이는 Word(한국어판 포함)에 대응 기능이 없는 한국 고유 기능**이다(반증 검색까지 통과). 부호만 삽입하거나 '추가하거나 바꿀 내용'(제안 대체 텍스트)을 부호와 함께 삽입하는 두 방식 — 교정 표시 1건이 "지적 + 제안 텍스트"를 함께 담는 단위라는 국내 선례로, 우리의 "코멘트 vs 수정 제안" 이원 구조와 동형이다.
- 주의: 변경 추적은 Windows판 한정(맥용 한/글 2022에는 부재 보고). 또한 "한국어 사용자가 X를 기대한다"류 서술은 기능 존재로부터의 해석이며 사용자 조사로 실증된 것은 아니다.

**설계 시사점**: 사용자가 요청의 출발점으로 삼은 "원고지 교정" 은유가 한국어 도구 생태계에 소프트웨어 관례로 확립되어 있다. 편집 데스크 UI의 어휘·액션을 한/글 관례(교정 부호, 적용 후 다음으로 이동, 4단계 보기)에 정렬하면 감독자 학습 비용이 0에 가까워진다.

## 2. 공백 영역에서 회수된 패턴 [전부 검증]

- **Apple Pages — forced resolution**: 미결 변경을 전부 수락/거부하기 전에는 변경 추적을 끌 수 없다(몰래 버리기 불가, 폐기는 명시적 'Reject All' 선택으로만). 해소 granularity는 3계층(변경 1건 hover / 선택 범위 / 문서 전체). → "미결 제안이 남은 배치 PR은 병합하지 않는다" 규칙의 상용 선례로 §5.4에 반영.
- **Acrobat**: ① 주석을 문서와 분리된 FDF/XFDF 파일로 직렬화하는 오버레이 데이터 모델 — "주석 레이어 별도 저장 + 동기화" 아키텍처의 상용 선례. ② PDF 교정 주석 중 **텍스트 편집(삽입/삭제/대체)만 선별해 원본 Word 문서로 내보내 소스를 수정**하는 파이프라인이 존재하며, 최종 수락은 인간이 건별로 확인한다(Apply/Apply All/Discard). 단 반자동이며 태그드 PDF(PDFMaker 생성)·Windows 전제. → "기계 적용 가능한 편집과 일반 의견을 구분하고, 자동 반영이라도 승인은 건별" — 우리의 코멘트 유형 분류 + 승인 게이트와 동형.
- **Figma**: 캔버스 좌표에 고정되는 댓글 핀(hover 미리보기), 전용 댓글 사이드바(파일 전체 주석 + 소속 페이지 표시), resolved 시 양쪽에서 숨김 + "Show resolved" 토글(un-resolve 가능). 주의: 핀이 요소 이동을 따라가지 않는 좌표 앵커라서, 텍스트 교정에는 콘텐츠 앵커링으로 개선이 필요하다 — 우리 앵커 설계(§10)가 이미 그 방향이다.
- **GitBook change request**: 리뷰 단위는 개별 코멘트가 아니라 **스페이스 본문 전체의 사본(브랜치)** — 여러 페이지 편집 묶음이며, 공식 문서가 GitHub PR과 같은 브랜칭 모델임을 명시한다. diff는 split-view 기본 + 인라인 전환, '전체 맥락 / 변경 페이지만' 선택. **[반박·정정]** 초판 보고서에 실었던 "병합 후 롤백 불가"는 검증에서 0-3으로 기각되었다 — 병합 롤백 가능성은 미확정으로 취급하고, 우리 모델의 비교 우위 논거로 쓰지 않는다.
- **PubPub / MIT Press 커뮤니티 리뷰**: 리뷰 단위는 "공개된 원고 전체 + 정해진 리뷰 기간(8~9주)". 조사된 3개 프로젝트 중 2개의 저자들이 주석을 실시간 처리하지 않고 **기간 종료 후 일괄 반영**했다("피드백이 가라앉고 숙성되기를 원했다"). → "주석 수집은 실시간, 소스 반영은 배치"(§5.6)가 학술 교재 교정의 실전 관행. (Data Feminism 저자들은 대조적으로 수시 반영 — 관행이 유일하지는 않다.)
- **Gemini in Docs**: AI 제안마다 개별 Accept/Reject 컨트롤 + 'Show suggestion' 순회 검토 + 선택적 Accept all — AI 제안을 기존 변경 추적 파이프라인의 개별 수락 단위로 흘려보내는 패턴. LLM 교정 제안을 "건별 수락 가능한 suggestion"으로 렌더링하는 우리 설계의 선례. (Copilot in Word·Notion AI의 대응 디테일은 검증 통과 주장이 없어 미확인.)

## 3. 편집 이력의 알갱이 크기 — coalescing 선례

"같은 문구를 여러 번 고치면 커밋도 여러 개인가?"에 대한 업계의 답은 일관된다: **아니다.** 검증된 선례들이 수렴하는 것은 **이원(二元) 모델**이다:

1. **비공개·작업 중 구간**(에디터 버퍼, 세션, PR 열림 중)에서는 연속 수정을 시간·행위·세션 경계로 자유롭게 압축하고 최종(또는 첫) 상태만 남긴다.
2. **공개 구간**(병합된 이력, 위키 리비전)에서는 이력을 압축·삭제하지 않고 append-only로 유지하되, **표시(annotation)와 뷰 필터링**으로 검토 노이즈를 관리한다.
3. 알갱이의 정의 기준은 두 구간 모두 기술적 크기가 아니라 **의미 단위** — "한 번에 통째로 되돌릴 수 있는 행위"(Apple), "검토 필요성·분쟁 가능성"(위키백과).

### 3.1 에디터 undo와 CRDT [검증]

- **NSUndoManager(Apple)**: 실행 루프 한 사이클 안의 모든 undo 연산을 자동으로 한 그룹으로 수집하고, undo 시 그 사이클 전체가 한꺼번에 되돌려진다. 그룹의 설계 기준은 "전체가 한 번에 되돌릴 수 있는(entirely revertible) 행위". 실제 텍스트 편집은 `breakUndoCoalescing`으로 여러 사이클에 걸친 타이핑을 더 굵게도 병합한다 — 프레임워크가 기본 압축 경계를 정해 주는 선례.
- **Blender 설계 논의**: 알갱이 옵션 세 가지(키 입력별 / 행위 유형별 / 타이머)를 열거했고, 타이머 방식의 단점을 "타이핑 속도가 임계 근처면 어떤 편집이 묶일지 예측 불가"라고 기록했다. **[반박·정정]** 초판 보고서의 "위치·의미 경계 방식이 채택됨"은 0-3으로 기각 — 검증된 것은 시간 기반의 예측 불가능성 비판까지다. (§5.5의 "시간 임계값 자동 확정을 쓰지 않는다"는 이 검증된 근거로 충분히 서고, 국소성 원칙의 귀속은 아래 Yjs로 교체한다.)
- **Yjs(CRDT) Item 병합**: 연속 타이핑된 문자열은 문자마다 논리 시계가 증가함에도 단일 Item으로 병합 저장되며, 병합 조건은 "같은 clientID·순서대로 삽입·삭제 상태 균일" — **같은 주체의 끊기지 않은 연속만 병합하고, 흐름이 끊기면 다시 분할**된다. "굵게 저장하되 필요하면 세분화 가능"한 조건부·가역적 압축의 공학 선례. → §5.5 확정 규칙 ③(개입 대상이 다른 앵커로 이동하면 확정)의 귀속 근거.
- **압축 모드의 코드화**: NONE/FIRST/LAST/중복 제거 모드가 라이브러리에 코드화되어 있고 근거는 "변경은 많지만 보존 가치는 첫/끝 상태뿐인 경우"(JS-UndoManager — 단 2018년 아카이브된 소규모 라이브러리라 분류 예시로만 유효). 같은 문구 반복 다듬기는 정확히 LAST 모드다.
- **y-partykit 영속화**: 권장 모드(snapshot)는 세션 중 업데이트를 쌓다가 **마지막 클라이언트가 끊기는 시점에 하나의 스냅숏으로 병합**, 최종 상태만 보존 — "세션 종료를 압축 경계로" 쓰는 관행. 우리의 "병합 요청 시점 autosquash 1회"와 같은 자리다.

### 3.2 위키백과 리비전 모델 [검증] — 공개 구간의 노이즈 관리

- **minor edit 플래그는 압축이 아니라 표시 장치**: 사소한 편집도 전부 리비전으로 남되, 플래그로 최근 변경·주시문서에서 필터링해 검토 부담을 줄인다. 판정 기준은 크기가 아니라 **"검토가 필요 없고 분쟁 대상이 될 수 없는가"** — 의미에 영향을 주면 단어 하나여도 minor가 아니다. → 공개 이력에서는 압축 대신 '검토 필요성 등급 + 필터링'이 검증된 대안.
- **rollback vs undo — 되돌리기의 두 알갱이**(2-1 표결 포함): rollback은 **같은 편집자의 연속 리비전 묶음**을 한 클릭에 되돌리고(권한 필요), undo는 단일 편집을 이후 편집 보존하며 되돌린다. "연속 수정 묶음"을 저장 단위가 아니라 **조작(되돌리기·검토) 단위**로 정의하는 선례.
- **되돌리기도 이력에 추가된다**: rollback은 결과가 새 리비전으로 기록되고 되돌려진 편집도 남는다 — §5.3(revert 커밋, force-push 금지)과 같은 철학. **[반박·정정]** 단 "위키백과 이력은 절대 삭제되지 않는다"는 일반화는 0-3 기각(RevisionDelete/oversight 같은 관리적 예외 존재). append-only는 통상 편집 흐름에 한정해 성립한다.
- **봇 노이즈는 뷰에서 거른다**: bot rollback과 봇 플래그 편집은 최근 변경 목록에서 기본 숨김되지만 데이터베이스와 페이지 히스토리에는 전부 남는다. → AI 교정 커밋이 대량일 때 이력 자체를 줄이지 말고 대시보드 뷰에서 `actor:` 라벨로 필터링하는 우리 설계와 일치.
- **WP:COSMETICBOT**: 실질 변경 없이 표면만 바꾸는 편집을 단독 리비전으로 만드는 것을 강하게 제한하고, 실질 변경이 있는 편집에 **동승**시키도록 요구한다("should not usually be done on their own"). → 렌더링 결과에 영향 없는 소스 정리(공백·마크업 정규화)는 단독 커밋 금지, 실질 교정 커밋에 동승 — §5.4에 반영.
- **동시 편집 충돌의 기준선**: MediaWiki는 diff3 자동 병합을 먼저 시도하고 **"같은 몇 줄"을 건드린 경우에만** 충돌을 발생시키며, 실패 시 3단 화면(상대 저장본/diff/내 미저장본)에서 사람이 병합한다. → 두 교정자가 다른 문단을 고치면 자동 병합, 같은 문장일 때만 감독자 판단(§11-4)의 검증된 기준선.

### 3.3 중간 시행착오의 보존 가치 [검증] — AI 학습 관점의 답

- **Yjs는 삭제 메타데이터를 의도적으로 폐기한다**: 언제·누가 지웠는지 기록하지 않고, GC 기본값에서는 삭제된 콘텐츠 자체가 통째로 버려진다. 보존하려면 명시적으로 GC를 꺼야 한다(opt-in).
- 즉 "중간 시행착오 상태는 기본적으로 영구 보존 가치가 없다"가 공학적 합의이고, **보존은 기본값이 아니라 명시적 opt-in 채널**로 설계해야 한다. §5.5의 "확정 상태만 이벤트 레코드에 적재, 협의 스레드의 기각 대안은 별도 보존"이 정확히 이 구조다. 다만 중간 궤적이 스타일 학습에 실제로 유의미한 신호를 주는지의 실증 근거는 업계에 없다(열린 질문 §4).

### 3.4 Git 규범 [미검증 — 설계는 유지]

초판에 실었던 Git 이력 규범(리베이스 황금률, force-push의 리뷰 앵커 파괴, thoughtbot autosquash 관행, squash merge의 작성자 정보 소실)은 **두 번의 검증 사이클에서 살아남은 1차 출처가 없다** — 수집 단계 자료와 일반 지식에 기댄 상태다. 다만:

- §5.5의 "병합 전 다듬기 / 병합 후 불변" 이원 규칙 자체는 §3.1~3.2의 **검증된 이원 모델과 구조적으로 동형**이어서 Git 출처 없이도 선다.
- fixup/autosquash·merge commit 세부는 표준 도구 동작이라 실무 채택 위험이 낮다. 단 "force-push가 리뷰 코멘트를 outdated로 만드는 정확한 규칙"은 GitHub 공식 문서로 확인할 것(열린 질문).

### 3.5 버전 기록 UX [미검증]

Google Docs의 "자동 세션 압축 + 이름 있는 버전" 2층 구조와 Notion 스냅숏 주기는 검증 통과 주장이 없다. 감독자 이력 뷰의 "기본은 배치 단위 압축, 펼치면 교정별, 마일스톤은 Git 태그" 설계는 유지하되 근거는 §3.1~3.2의 검증 선례로 갈음한다.

## 4. 두 사이클 후에도 남은 공백 (열린 질문)

세션 한도가 아니라 **검증을 통과한 출처를 확보하지 못해** 남은 것들이다:

1. **자동화 경제성의 정량 근거**: Dependabot/Renovate grouped updates 설정·도입 배경, changesets 배치 릴리스, stacked PR 리뷰 단위 철학, webhook vs 폴링 vs cron digest의 비용·지연 트레이드오프. §5.6의 다이제스트+사전 필터 설계는 자체 논리로 서지만 업계 정량 선례는 미확보.
2. ~~**블록 ID·주석 동기화 기술 검증**~~ → **2026-07-18 자체 스파이크로 해소**: ① 콘텐츠 해시 앵커 생존성 실측(Pandoc 3.9, 6개 수정 시나리오) — 무수정 문단은 삽입·순서 교체·서식 변경에서 100% 유지, 수정·분할된 문단만 자체 ID 변경(의도된 동작 — 인용문+문맥으로 재앵커). ② Hypothesis 라이브 API 실측 — `GET /api/search`의 `uri` 필터와 `TextQuoteSelector{exact, prefix, suffix}` 구조 확인: 우리 `editorial-anchor`와 동형. 기성 Hypothesis↔GitHub 동기화 사례는 미발견 — 자체 구현(`platform/scripts/editorial.py`)으로 진행.
3. GitHub force-push 후 리뷰 코멘트 outdated 처리의 정확한 앵커 매칭 규칙, 사람 편집이 봇 제안을 supersede하는 처리의 직접 선례.
4. Copilot in Word·Notion AI의 부분 수락/재프롬프트/로딩 상태 패턴, Google Docs 세션 묶음의 정량 파라미터.
5. 중간 시행착오 궤적이 스타일 학습에 주는 실증적 가치(선례는 "기본 폐기, opt-in 보존" 관행만 답함).
