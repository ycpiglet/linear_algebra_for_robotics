# 문서 편집·리뷰 도구 딥 리서치: 필요 기능 카탈로그

> 문서 성격: `EDITING_SYSTEM_PLAN.md`(협업 교정·편집 시스템 기획서)를 뒷받침하는 조사 보고서
>
> 조사일: 2026-07-17 · 방법: 5개 각도 병렬 웹 조사 → 25개 소스 정독 → 124개 주장 추출 → 상위 25개 3중 교차 검증(22 확인 / 3 반박)
>
> 표기: 본문에서 **[검증]**은 3인 교차 검증을 통과한 사실, **[참고]**는 1차 출처에서 추출했으나 교차 검증 단계에 오르지 않은 내용이다. 난이도 평가와 구현 스케치는 검증된 사실이 아니라 본 저장소 아키텍처(정적 사이트 + Git + GitHub PR + 주석 레이어)에 대한 설계 판단이다.

## 0. 요약: 이식할 4개 핵심 패턴

1. **표시 밀도의 2축 분리** — Word는 "무엇을 추적하나"(전원/본인만)와 "얼마나 자세히 보여주나"(Simple/All/No Markup/Original)를 분리한다. 감독자에게는 기본적으로 저밀도 신호(여백 표시선)만 보여주고, 원할 때만 상세를 펼치게 하는 것이 인지 부하 관리의 표준이다. **[검증]**
2. **제안의 3중 뷰** — Google Docs는 제안을 "인라인 표시 / 모두 수락된 상태 미리보기 / 모두 거절된 상태 미리보기"의 3개 뷰로 제공한다. 우리 아키텍처에서는 main 배포 vs PR review artifact의 이중 빌드가 이를 자연스럽게 실현한다. **[검증]**
3. **역할이 모드를 강제** — Overleaf는 리뷰 권한자가 Reviewing 모드로 전환하면 변경 추적이 자동으로 켜지고, 권한이 낮으면 모드를 바꿀 수 없다. "감독자·에이전트의 수정은 항상 제안(PR)으로 기록된다"를 개인의 조심성이 아니라 시스템 정책으로 만드는 선례다. **[검증]**
4. **객체(블록) 단위 독립 버전** — Curvenote는 문단·수식·그림 각각이 독립 스냅숏 이력을 가지며, 수식은 별도 객체로서 LaTeX 편집기로 전환해 수정한다. 수식 밀도가 높은 이 교재에서 차별화 가치가 가장 큰 기능이고, 구현 난이도도 가장 높다. **[검증]**

그리고 결정적인 실전 선례 하나: **Quartorium** — Quarto(.qmd) + Git 위에서 비개발자가 코드를 보지 않고 워드처럼 편집·댓글하면 별도 브랜치에 자동 커밋되고, 저자가 원클릭 수락/거부하는 오픈소스 협업 편집기(TipTap + Node/Express + Git + Quarto CLI). 우리 아키텍처와 정확히 동형이므로 Phase 2 설계 전 반드시 스택과 왕복 변환 방식을 분석할 것. **[참고]**

## 1. 기능 카탈로그 (a)

각 항목: 무엇인가 → 대표 구현 → UX 디테일 → 우리 아키텍처에서의 난이도와 실현 스케치.

### 1.1 변경 추적과 마크업 표시

| 요소 | 대표 구현 | 검증 상태 |
|---|---|---|
| 추적 범위 선택 (전원 / 본인만) | Word: For Everyone / Just Mine | [검증] |
| 표시 밀도 4단계 | Word: Simple Markup(여백 빨간 세로선만) / All Markup(검토자별 색상) / No Markup(반영 후) / Original(원본) | [검증] |
| 제안 표시 위치 | CKEditor 5: 우측 사이드바 주석 풍선, 클릭 시 확장 | [참고] |

**난이도: 낮음~중간.** PR diff가 All Markup에, gutter 표시가 Simple Markup에 대응한다. 구현 스케치: 리뷰 빌드에서 "현재 페이지에 열린 제안이 있는 문단"에 여백 인디케이터만 표시(저밀도 기본값), 클릭하면 해당 제안 상세(diff·코멘트)가 우측 패널로 펼쳐지는 2단 구조. 작성자별 색상은 `actor:supervisor/editor/agent` 라벨과 매핑.

### 1.2 수락/거부 레퍼토리

**[검증]** Word의 표준 레퍼토리: ① 순차 검토(Accept and Move to Next) ② 개별 처리(변경 위 컨텍스트 메뉴) ③ 일괄 처리(Accept All) ④ 복합 동작(Accept All and Stop Tracking) ⑤ Next/Previous 탐색. (주의: "범위 선택 → 일괄 수락"이 Overleaf에 있다는 주장은 검증에서 반박됨 — 표준 패턴으로 간주하지 말 것.)

**난이도: 낮음.** GitHub PR의 suggestion(Apply suggestion)·배치 커밋·리뷰 스레드가 네이티브로 대응. 감독자용 대시보드에 필요한 최소 조합은 **"다음 제안으로 이동 + 수락" 순차 루프**와 **"전체 수락"** 두 가지다. 키보드(j/k 이동, a 수락, r 거절)까지 얹으면 워드프로세서 검토 경험과 등가가 된다.

### 1.3 제안(suggestion) 모드

- **[검증]** Google Docs: 3중 뷰(SUGGESTIONS_INLINE / PREVIEW_SUGGESTIONS_ACCEPTED / PREVIEW_WITHOUT_SUGGESTIONS). 제안은 별도 레이어가 아니라 문서 트리 안 인라인 주석으로 저장.
- **[검증]** Overleaf: Editing/Reviewing/Viewing 모드 토글, Reviewing 진입 시 Track Changes 자동 활성화, 권한 낮은 사용자는 모드 변경 불가.
- **[참고]** Notion: ••• 메뉴 → Suggest edits, 활성화 시 페이지 상단 "Suggesting" 배지. **댓글 권한(Can comment)만 있어도 제안 가능** — 편집 권한 없는 검토자가 안전하게 수정안을 내는 권한 계층.

**난이도: 낮음(정책) + 중간(UI).** 정책 측면은 기획서 §5 그대로: 에이전트·편집자는 PR만, 병합은 감독자. "Suggesting 상태 배지"에 해당하는 것은 리뷰 빌드 상단의 "리뷰 모드 — 이 화면의 수정은 제안으로 기록됩니다" 띠. Notion의 권한 계층은 "GitHub 계정에 쓰기 권한이 없어도 이슈(제안)는 낼 수 있다"로 이미 대응된다.

### 1.4 댓글 스코핑과 수명주기

**[검증]** Notion의 완결된 패턴 집합:
- **2계층 스코핑**: 페이지 전체 댓글(상단 Add a comment)과 블록 단위 댓글(⋮⋮ 핸들 메뉴 / 호버 💬 / 텍스트 선택 팝업 / 단축키)의 분리. 진입점이 복수라는 점이 중요하다.
- **마진 표시**: 인라인 댓글은 해당 텍스트 오른쪽 여백에, 긴 스레드는 접힘. Default/Minimal 표시 밀도 선택.
- **수명주기**: 페이지 상단 💬로 여는 댓글 패널에서 사람·해결상태별 필터, 호버 후 ✔️로 해결(resolve), 해결된 스레드 재오픈. (주의: "••• 오버플로 메뉴로 해결"은 반박됨 — ✔️ 아이콘 경로가 사실.)

**난이도: 스코핑·마진 표시 중간, resolve/reopen 낮음(GitHub 리뷰 스레드 resolve와 1:1).** 구현 스케치: 주석 레이어(Hypothesis 또는 자체 패널)에서 (i) 텍스트 선택 → 문장 단위 코멘트, (ii) 객체 호버 → 💬 아이콘 → 블록 단위 코멘트, (iii) 페이지 상단 → 페이지 전체 의견의 3개 진입점을 제공. 페이지별 "열린 코멘트 n건" 배지와 필터 가능한 패널은 GitHub API 조회로 대시보드에 구현(기획서 1-4).

### 1.5 버전 이력·복원·비교

**[참고]** CKEditor 5의 개념 구분이 유용하다: **Track Changes는 "미래에 적용될 변경"(협업 제안)**, **Revision History는 "이미 적용된 변경"(감사 추적: 스냅숏 생성·비교·복원)** — 목적이 다른 두 기능을 함께 켜야 완전한 워크플로우가 된다.

**난이도: 낮음.** 우리에겐 이미 둘 다 있다 — 전자가 PR, 후자가 git 이력. 부족한 것은 감독자용 노출뿐이다: 페이지 하단 "최근 변경"(기획서 2-3), 대시보드의 [이 편집 되돌리기](revert PR), 그리고 "두 시점 비교"는 태그된 빌드 간 링크로 제공.

### 1.6 블록 조작 UX (호버·드래그·슬래시)

- **[검증]** Notion: 블록 호버 시 ⋮⋮ 드래그 핸들 + 메뉴, 슬래시 명령은 삽입·서식·블록 유형 변환(/turn 계열)을 하나의 팔레트로 통합, @-멘션 시 백링크 자동 생성·노출.
- **[검증]** Tiptap Notion형 템플릿: 블록 드래그 재배열·핸들 컨텍스트 메뉴·슬래시 명령·수식 블록·구문 강조 코드 블록·이미지 드래그 업로드가 상용 템플릿 수준으로 존재. **단, 표는 기능 목록에 없고 로드맵 항목이며, 실시간 협업·AI 기능은 서버 JWT + 유료 클라우드 의존.**

**난이도: 읽기 화면의 호버 어포던스는 낮음, 편집기 껍데기는 중간, 실시간 협업은 높음이자 불필요.** 시사점 두 가지: (1) 렌더된 읽기 화면에도 "객체 호버 → 액션 아이콘(💬 코멘트 / ✏️ 수정 제안 / 🔗 링크 복사 / 🕘 이력)"이라는 Notion식 어포던스를 얹을 수 있고 이것이 체감 편의성의 핵심이다. (2) 실시간 공동 편집(라이브 커서)은 비동기 PR 모델과 상충하고 서버·비용 부담만 크므로 **도입하지 않는 것이 합리적**.

### 1.7 찾기·확대·탐색

조사에서 교차 검증까지 오른 항목은 없었으나, 기존 인프라와의 대응을 정리한다: 사이트 전역 검색(내비바 오버레이)·페이지 내 목차·읽기 진행률은 이미 있다. 추가 가치가 있는 것은 (i) 그림·표 클릭 확대(라이트박스) — Quarto 확장으로 난이도 낮음, (ii) 수식 확대·원본 보기 — 수식 호버 툴바(§2)로 흡수, (iii) 검색 결과에서 문단 앵커로 직행 — 문단 ID 필터(기획서 1-1)와 결합.

### 1.8 주석 레이어의 앵커 지속성

**[참고]** Quarto 사이트에 `comments: hypothesis: true` 한 줄로 주석 레이어가 켜지고, 로컬 렌더·비공개 배포·공개 배포 모두에서 동작하며, **텍스트 앵커링 기반이라 재렌더링해도 주석이 유지**되고 앵커 텍스트가 완전히 삭제된 경우에만 사라진다. 반면 계정 가입 필요, 공개/비공개 설정 UX 혼란("Post to only me" 오동작 관찰) 등 온보딩 마찰이 보고되어 있다. Giscus는 페이지 하단 댓글용이라 문장 단위 교정에는 부적합하다.

**시사점**: Phase 0에서 Hypothesis를 켜되, 편집팀 온보딩 문서(그룹 가입 → 비공개 그룹 선택까지 스크린숏 3장)를 함께 제공해 마찰을 흡수할 것. 장기적으로 재렌더링 내성은 "인용문+문맥" 앵커의 타당성을 실증하는 근거다.

## 2. 객체(수식·표·이미지·코드) 단위 편집·주석·이력 (b)

### 2.1 선례에서 확인된 것

- **[검증]** Curvenote: 문단·수식·그림·코드 셀 각각이 독립적인 불변 버전 스냅숏 이력을 가짐. 수식은 별도 객체로 취급되어 필요 시 LaTeX 수식 편집기로 전환(drop into)하며 자동 번호·상호참조·참조 호버 프리뷰 기본 제공. (주의: Curvenote는 "객체 버전 관리" 선례로만 인용 — "인라인 댓글+제안 협업 제공" 주장은 반박됨. 제품 축도 MyST/Git 기반으로 전환 중.)
- **[참고]** InCopy/InDesign 편집 워크플로우: 문서를 객체 단위 구성요소(제목·본문·그래픽·캡션)로 분할해 서로 다른 편집자에게 할당(assignment)하고, 체크아웃 방식으로 동시 편집 충돌을 차단하며, 레이아웃 편집과 콘텐츠 편집의 역할을 분리.

### 2.2 우리 아키텍처에서의 설계

이 저장소는 이미 객체 ID의 절반을 갖고 있다. Quarto 라벨(`#eq-…`, `#fig-…`, `#tbl-…`, 코드 청크 라벨)이 곧 안정적 객체 ID이고, 라벨 없는 문단만 해시 기반 ID(기획서 1-1)로 보충하면 된다.

**객체 호버 툴바 (난이도: 중간, 체감 효과 최대).** 렌더된 페이지에서 수식·표·그림·코드 블록에 호버하면 우상단에 아이콘 그룹이 페이드인:

| 아이콘 | 수식 | 표 | 그림 | 코드 |
|---|---|---|---|---|
| 💬 코멘트 | 이 수식에 스레드 | 동일 | 동일 | 동일 |
| ✏️ 수정 제안 | **LaTeX 원본이 채워진 편집창** → 제출 시 이슈/PR | 셀 단위 구조 편집창(마크다운 표로 왕복) | 대체 이미지 업로드 or 캡션·대체텍스트 수정 | 코드 원본 편집창 |
| 🔗 링크 복사 | `#eq-…` 앵커 | `#tbl-…` | `#fig-…` | 청크 앵커 |
| 🕘 이력 | 이 객체의 변경 이력 | 동일 | 동일 | 동일 |
| ⤢ 확대 | 수식 확대 표시 | 전체 폭 보기 | 라이트박스 | — |

핵심은 ✏️에서 **객체의 소스가 미리 채워진 편집창**이 뜬다는 것: 감독자는 렌더된 수식을 보다가 바로 그 자리에서 TeX 한 항을 고치고 제출하면, 시스템이 해당 `.qmd`의 해당 라벨 위치로 매핑해 PR을 만든다. 수식 편집창은 MathLive류 시각 편집기 또는 실시간 미리보기가 붙은 TeX 입력창으로 시작한다. Tiptap 템플릿조차 표를 아직 지원하지 않는다는 사실이 시사하듯 **표의 WYSIWYG 편집이 가장 어렵다** — 표는 1차로 "마크다운 표 텍스트 편집 + 미리보기"로 시작하고 셀 단위 UI는 후순위로 미룬다.

**객체 단위 이력 (난이도: 높음, 가치 큼).** 등가 구현: 객체 라벨을 키로 `git log -L`(라벨이 포함된 범위 추적) 또는 편집 이벤트 레코드(기획서 §6)의 `object_label` 필드 조회. 완전한 Curvenote식 블록 버전 DB를 만들 필요는 없다 — "이 수식이 언제 누구에 의해 어떻게 바뀌었나"를 시간순으로 보여주는 조회 뷰면 감독자 요구는 충족된다.

**편집 이벤트 스키마 확장.** 기획서 §6 레코드에 두 필드를 추가한다:

```yaml
object_type: equation | table | figure | code | paragraph | block
object_label: eq-kalman-gain      # Quarto 라벨 또는 문단 해시 ID
```

이로써 "감독자가 수식을 어떻게 고치는 경향이 있는가"(예: 표기 관례, 첨자 스타일, 유도 단계 분할 습관)를 객체 유형별로 분리 학습할 수 있다 — 문체 학습과 수식 표기 학습은 원료가 다르기 때문에 이 분리가 중요하다.

## 3. 협의형 코멘트와 AI 응답 루프 (c)

### 3.1 확인된 상호작용 패턴

- **[검증]** Google Docs API `writeMode=SUGGEST`: API로 넣는 모든 편집이 직접 반영 대신 "제안"으로 기록되고, Accept/Reject/DeleteSuggestionRequest로 프로그래밍 관리(개발자 프리뷰). **"AI가 제안 생성 → 사람이 수락/거절" 루프의 직접적 상용 선례**이며 GitHub PR 리뷰와 구조적으로 동형.
- **[참고]** Gemini in Docs: AI 제안이 채팅창이 아니라 **본문 인라인**에 렌더되고, 개별 수락 / 전체 수락 / 전체 거절로 처리. 프롬프트 UI는 하단 바 → 선택 시 플로팅 Refine 툴바 → 긴 응답은 사이드 패널로 이동하는 계층적 배치.
- **[참고]** Copilot in Word: **유지(keep) / 폐기(discard) / 후속 프롬프트로 계속 다듬기(refine)**의 3선택 루프가 표준. 공동 편집 문서에서는 제안 미리보기를 채팅에 먼저 보여주고 승인 전에는 문서에 반영하지 않음(승인 게이트).
- **[참고]** DraftView: PR을 구글독스형 렌더 화면으로 보여주고, 거기서 작성된 제안을 GitHub 네이티브 suggested change로 역동기화 — 비개발자 리뷰어 문제에 대한 우리와 동일한 해법의 상용 구현.

### 3.2 설계: 코멘트 유형 분류와 응답 계약

사용자 요구("이런 표현보다는 어떤 표현이 좋겠어", "혹시 이런 스타일로 수정 가능할까")는 지시가 아니라 **협의**다. 코멘트를 4유형으로 분류하고 유형별 에이전트 응답 계약을 다르게 한다. 유형은 감독자가 고르게 하지 말고(마찰), 에이전트가 문면에서 분류하되 확신이 낮으면 되물어본다.

| 유형 | 예 | 에이전트 응답 계약 |
|---|---|---|
| 지시(directive) | "‘업데이트’를 ‘측정 갱신’으로" | 바로 수정 PR. 답글로 diff 링크 |
| 구체 제안(suggestion) | 수정문을 직접 써 줌 | 소스 매핑 후 그대로 PR (에이전트는 형식만 정리) |
| **협의(consultative)** | "이 표현보다 나은 게 없을까?", "이런 스타일 가능해?" | **바로 고치지 않는다.** 답글로 대안 2~3안을 근거(매뉴얼 규칙·용어집·기존 용례)와 함께 제시 → 감독자가 답글/이모지로 선택 → 선택안만 PR |
| 방향(direction) | "이 절 전체가 너무 건조해" | 대표 문단 1~2개만 시범 수정한 미리보기 PR → 승인 시 절 전체 적용 PR (Copilot의 refine 루프 + 승인 게이트) |

협의·방향 유형의 왕복 기록(제시된 대안, 감독자의 선택, 기각된 안)은 지시 유형보다 **학습 신호로서 가치가 높다** — "무엇을 골랐는가"와 "무엇을 버렸는가"가 쌍으로 남기 때문이다. 편집 이벤트 스키마의 `instruction`에 대화 스레드 전체를 연결한다.

### 3.3 UI 배치

Gemini의 계층적 배치를 이식한다: 기본은 텍스트 선택 시 뜨는 소형 팝업(코멘트/수정 제안/링크 복사), 협의형의 긴 왕복은 우측 패널의 스레드 뷰로 승격, 절 단위 방향 지시는 페이지 상단 페이지 코멘트로. 모든 AI 제안은 승인 게이트 뒤에 있다(자동 병합 없음 — 기획서 §5.1과 일치).

## 5. 조사의 한계와 열린 질문

**커버리지 공백** — 다음은 검증 단계에 오르지 못해 이 카탈로그가 다루지 않는다: 한글(HWP)·Apple Pages의 리뷰 기능(한국어 감독자의 UX 기대치 형성 요인이므로 후속 조사 가치 있음), Confluence/Dropbox Paper/Coda/GitBook/Obsidian Publish, Adobe Acrobat 주석·Figma 댓글 핀, Typst/PubPub. AI 도구의 프런트엔드 UX는 [참고] 수준 근거만 확보.

**반박된 주장 (인용 금지)** — ① Notion 댓글의 ••• 오버플로 메뉴 수명주기(0-3, 해결은 ✔️ 경로가 사실) ② Curvenote의 인라인 댓글+제안 협업(1-2, 객체 버전 관리 선례로만 인용) ③ Overleaf의 범위 선택 일괄 수락 3단계 워크플로우(0-3).

**시간 민감성** — Google Docs SUGGEST writeMode는 개발자 프리뷰. Tiptap 무료 플랜은 2025-06 폐지($49/mo~). Curvenote는 MyST/Git 기반으로 제품 축 전환 중.

**열린 질문** — ① Gemini/Copilot의 부분 수락·재프롬프트 등 세부 UX 디테일 ② HWP 변경 추적·메모 관례 ③ `.qmd` 블록 ID 부여의 검증된 기법(Pandoc `{#id}` 속성·Quarto 라벨 활용 — §2.2의 접근은 설계 가설) ④ Hypothesis류 앵커와 GitHub 리뷰 스레드의 양방향 동기화 가능성(DraftView가 부분 선례).
